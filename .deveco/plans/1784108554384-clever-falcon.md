# ECOS 架构审视 — 偏差分析与分Sprint迭代回补计划

## Goal

对照17份原始系统设计文档，识别所有不满足原设计方案的内容，经用户确认方向后，制定分Sprint迭代回补计划，逐步将工程实现对齐到设计目标。

## 用户确认的偏差处理方向

### 🔴 8项重大偏差

| # | 偏差项 | 用户决策 |
|---|--------|---------|
| 1 | 微服务→单体 | **立即启动微服务拆分** |
| 2 | Kafka事件总线缺失 | **引入Kafka** |
| 3 | Database Schema未隔离 | **拆分Schema** |
| 4 | CI/CD流水线缺失 | **延后处理（Sprint 6）** |
| 5 | K8S生产部署未实现 | **先Docker Compose过渡** |
| 6 | World Model引擎缺失 | **全部回补** |
| 7 | Saga/Outbox模式未实现 | **随微服务拆分一起实现** |
| 8 | 可观测性栈不完整 | **完善自研方案+Grafana** |

### 🟡 25项中等偏差：全部回补

### ⚠️ 重要架构调整说明

**原cognitive-engine（认知引擎）需拆分/重命名**：
- 原cognitive-engine当前承载：Agent配置/AIP/Agent Mesh/知识图谱/守卫栏/NLQ/诊断等AI相关功能
- **重命名为AI引擎（ai-engine）**：继续承载Agent/AIP/NLQ/守卫栏等AI基础设施
- **新增认知引擎（cognitive-engine）**：按设计文档11-World Model定义，专门承载6层认知架构：
  - L1 企业状态层
  - L2 因果模型
  - L3 场景模型
  - L4 模拟引擎
  - L5 策略引擎
  - L6 认知规划

### 执行规则

1. **6个Sprint自动连续执行**，不需要逐个Sprint等用户确认
2. **最终提交6个Sprint计划执行情况报告**，包含每Sprint的完成情况、遗留问题、偏差

## Sprint 1：微服务拆分基础 + Schema隔离 + Kafka引入（预计4-6周）

**目标**：将单体Fat JAR拆分为微服务架构基础，建立Schema隔离和事件总线

### 1.1 Schema拆分（优先执行）

在PostgreSQL同一实例内创建9个Schema，按**06-数据库设计说明书**的映射关系迁移表：

| Schema | 来源模块 | 核心表 |
|--------|---------|--------|
| ecos_identity | sysman | identity_tenant, identity_user, identity_role, identity_permission, identity_organization, identity_user_role, identity_role_permission |
| ecos_catalog | datanet | catalog_dataset, catalog_dataset_field, catalog_tag, td_datasource, td_data_resource, td_data_field, td_catalog_item |
| ecos_ontology | dccheng+ontology-engine | ontology, ontology_entity, ontology_property, ontology_relationship, ontology_action, ontology_rule, ontology_domain, ontology_version |
| ecos_object | workspace | object_instance, object_property, object_relationship, object_state_machine, object_timeline, object_version, ecos_object_links |
| ecos_workflow | buszhi | ecos_workflow_*, ecos_workflow_task, ecos_workflow_approval, ecos_workflow_log |
| ecos_rule | buszhi | ecos_dq_rule, ecos_dq_issue, ecos_cognitive_rule |
| ecos_agent | aimod+runtime-core+hermes | ecos_agent, ecos_agent_registry, ecos_mission, ecos_mission_task, agent_call_log, profile_config |
| ecos_knowledge | dccheng(kg部分) | knowledge_node, knowledge_edge, glossary |
| ecos_mission | gateway(worldmodel部分) | ecos_wm_goal, ecos_wm_causal_link, ecos_wm_scenario, ecos_goal_tracking |
| ecos_audit | sysman(audit部分) | audit_log, security_log, ecos_spans, ecos_token_usage |

**步骤**：
1. 创建Schema DDL脚本（Flyway迁移V47）
2. ALTER TABLE SET SCHEMA迁移每张表
3. 修改所有MyBatis Mapper XML中的表名引用
4. 修改所有Repository类中的表名引用
5. 更新application.yml中数据源配置

### 1.2 微服务拆分（核心任务）

**引擎重命名**：
- `cognitive-engine` → `ai-engine`（承载Agent/AIP/NLQ/守卫栏）
- 新增 `cognitive-engine`（承载World Model 6层认知架构）

**第一批拆分8个核心服务**：

| 服务 | 来源 | 端口 | 独立Boot |
|------|------|------|---------|
| api-gateway | 新建Spring Cloud Gateway | 8080 | ✅ |
| identity-service | sysman(IAM部分) | 18081 | ✅ |
| catalog-service | datanet | 18082 | ✅ |
| ontology-service | dccheng+ontology-engine | 18083 | ✅ |
| object-service | workspace | 18084 | ✅ |
| workflow-service | buszhi | 18085 | ✅ |
| agent-service | aimod+runtime-core(Agent部分)+hermes | 18086 | ✅ |
| knowledge-service | dccheng(KG部分)+runtime-core(KG部分) | 18087 | ✅ |
| ai-engine | 原cognitive-engine重命名 | 18088 | ✅ |

**每个服务**：
- 独立`@SpringBootApplication`
- 独立`application.yml`（独立数据源指向对应Schema）
- 独立Dockerfile
- 独立健康检查端点

**API Gateway（Spring Cloud Gateway）**：
- 替换当前Tomcat MVC网关
- 路由规则：`/api/v1/{domain}/**` → 对应微服务
- 认证过滤器（JWT验证）
- 限流过滤器

**服务发现**：Spring Cloud LoadBalancer + 简单配置（暂不引入Eureka，使用静态配置）

### 1.3 Kafka引入

**依赖**：spring-kafka

**Topic创建**（按设计文档03-HLD）：
- ecos.identity（UserCreated, RoleAssigned等）
- ecos.ontology（OntologyPublished, EntityAdded等）
- ecos.object（ObjectCreated, ObjectUpdated, RelationshipCreated等）
- ecos.workflow（WorkflowStarted, TaskCompleted等）
- ecos.agent（AgentStarted, ToolExecuted, AgentCompleted等）
- ecos.knowledge（KnowledgeCreated, KnowledgeLinked等）
- ecos.audit（所有审计事件）

**事件格式**：
```json
{
  "eventId": "uuid",
  "eventType": "ObjectCreated",
  "timestamp": "2026-07-15T10:00:00Z",
  "source": "object-service",
  "payload": { ... }
}
```

**替换方案**：将Spring ApplicationEvent替换为Kafka Producer/Consumer

### 1.4 Saga/Outbox模式

**Outbox Pattern**：
- 每个服务添加`outbox_event`表
- 业务操作+事件写入同一事务
- CDC线程轮询outbox表，投递到Kafka

**编排式Saga**：
- Saga Orchestrator服务
- 每步执行+补偿事务定义

## Sprint 4：World Model认知引擎 + Ontology DSL编译器（预计4-5周）

**重要说明**：原cognitive-engine已重命名为ai-engine（Sprint 1完成），本Sprint新增认知引擎(cognitive-engine)专门承载World Model。

### 4.1 新增cognitive-engine微服务

独立微服务，端口18089，Schema ecos_cognitive

### 4.2 World Model 6层引擎

**L1 企业状态层**
- RealtimeStateService：从Ontology Objects聚合实时状态快照
- State(t)数据模型

**L2 因果模型**
- CausalEdge实体（source, target, weight）
- CausalGraphService：因果图构建+查询
- 因果推理算法（规则-based + 统计相关性）

**L3 场景模型**
- ScenarioService：场景CRUD
- 预设场景库（10+场景模板）

**L4 模拟引擎**
- MonteCarloEngine：10K随机场景模拟
- State(t)→Action→State(t+1)状态转移

**L5 策略引擎**
- StrategyEngine：Goal→Recommended Strategy
- 策略候选生成+ROI评估

**L6 认知规划**
- CognitivePlanningEngine：Goal→Simulation→Optimization→Plan

### 4.3 World Model API
- POST /api/v1/scenarios
- POST /api/v1/simulations/run
- POST /api/v1/predictions
- POST /api/v1/optimizations
- POST /api/v1/strategy/recommend

### 4.4 Ontology DSL编译器
- YAML DSL定义
- Compiler Pipeline：DSL→Meta Model→DB Mapping+Graph Mapping+GraphQL Schema
- 版本管理：DRAFT→REVIEW→PUBLISHED→DEPRECATED→ARCHIVED

### 4.5 ObjectQL完整实现
- 完整语法+聚合查询
- ObjectQL→Graph Query→SQL→Unified Result

### 4.6 数据湖(Iceberg)
- Iceberg表格式 + MinIO Catalog
