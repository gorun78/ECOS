# 知识工作台重构计划

## Goal

将知识工作台从8个平铺Tab重构为双轨六分组架构：
- **轨道A**（平台自用知识）：元数据同步 → 血缘解析 → 本体对齐
- **轨道B**（智能体知识）：图谱同步 → 图谱探索 → 术语库 → 分类体系

新增闭环设计器、工作台配置、认知引擎配置3个Tab，拆分GraphExplorer（783→≤400行），提取类型体系+API封装层，补2个BE端点。

## Scope / Non-goals

### In scope
- P1: typesAndConstants.ts + knowledgeApi.ts + KnowledgeView瘦身
- P2: 6个分组Tab重组 + 4个新Tab + GraphExplorer拆分
- P3: 2个BE端点（KnowledgeSettings + RAG扩展） + 闭环验证

### Out of scope
- 认知引擎V1子引擎实现（已完成）
- AIPKnowledgeView.tsx / aiworkbench/KnowledgeView.tsx 清理（后续任务）
- 新增npm依赖
- 已有API路径签名变更
- 前端i18n全量补全

## Current State And Constraints

### Frontend现状
- `KnowledgeView.tsx` 554行，34个useState，9个直接fetch调用
- 8个Tab平铺无分组
- `ArchitectureTab` 36行纯静态
- `GraphExplorerTab` 783行超标
- `types.ts` 仅1个`MetadataAsset`接口
- 使用`authHeaders()/isOk()/payloadOf()`而非统一的`apiFetchData()`
- `GraphCanvas.tsx` 已存在于`src/components/`（527行）
- `glossary.ts` + `glossaryClient.ts` 已存在，使用`apiFetchData`
- `workbench.ts` 已定义`KGNode`/`KGEdge`/`GlossaryTerm`/`GlossaryFilter`等类型

### Backend现状
- 6个Controller已存在：KnowledgeGraphController(8端点)、KnowledgeApiController(3占位)、GraphSyncController(4端点)、GlossaryController(5端点)、ClassificationController(1端点)、CognitiveConfigController(4端点)
- KnowledgeApiController全是占位实现（sync→"queued"，query→空列表，index-status→零值）
- **6个前端调用的端点无后端**：`/api/ontology/mappings`、`/api/ontology/export`、`/api/integration/*`、`/api/lineage/*`
- GlossaryController路径是`/api/glossary`（非`/api/knowledge/glossary`）
- ClassificationController路径是`/api/catalog`（非`/api/knowledge/classification`）

### 关键约束
- **禁止新增npm依赖**
- **禁止tsx中直接fetch** — 必须走knowledgeApi.ts
- **禁止硬编码配置** — 统一从typesAndConstants读取
- **禁止改动已有API路径签名**
- **禁止删除现有Tab已工作的API调用逻辑**（Read-Modify-Write）

|---|---|---|
| `GET /api/knowledge/graph` | `GET /api/knowledge/graph` | 使用实际路径 |
| `GET /api/knowledge/nodes/{id}` | 同上 | 使用实际路径 |
| `GET /api/knowledge/search?q=` | 同上 | 使用实际路径 |
| `GET /api/knowledge/path?s=&t=` | 同上 | 使用实际路径 |
| `GET /api/knowledge/neighbors/{id}?d=` | 同上 | 使用实际路径 |

### 类型体系设计

`typesAndConstants.ts`整合：
- 新增：`KnowledgeAsset`、`KnowledgeGraphNode`、`KnowledgeGraphEdge`、`ClosedLoopConfig`、`KnowledgeSettings`
- 复用：从`workbench.ts`重导出`KGNode`/`KGEdge`/`GlossaryTerm`/`GlossaryFilter`/`Domain`
- 复用：从`knowledge/types.ts`迁移`MetadataAsset`
- 图标映射：`KNOWLEDGE_ICONS`集中管理
- 静态配置：`CHUNK_SIZE_OPTIONS`、`VECTOR_MODELS`

### KnowledgeView瘦身策略

将34个useState下放到各Tab：
- SyncTab内部管理：assets、syncLogs、isSyncingAll
- LineageTab内部管理：lineageNodes、lineageLinks、parserFormatSelected、rawPayloadInput、impactResult
- OntologyTab内部管理：ontologyMappings、availableTables、editingOntology、exportedMarkdown
- IndexTab内部管理：embeddingModel、chunkSize、overlap、vectorChunks、persistenceTab
- RagTab内部管理：queryInput、retrievedDocs、ragPrompt、llmOutput

KnowledgeView只保留：activeSubTab + 分组UI + Toast

### GraphExplorerTab拆分

`GraphCanvas.tsx`已存在于`src/components/`（527行），GraphExplorerTab已import它。所以不需要再抽Canvas渲染器——只需将GraphExplorerTab的交互逻辑精简（783→≤400行）：
- 抽离节点详情面板 → `components/NodeDetailPanel.tsx`
- 抽离路径分析弹窗 → `components/PathAnalysisModal.tsx`
- 抽离节点/边创建表单 → `components/CreateFormModal.tsx`

### Tab分组UI设计

用二级标签（非嵌套路由）实现分组：
```
📋 闭环设计
─── 知识准备（轨道A） ───
  元数据同步 | 血缘解析 | 本体对齐
─── 知识构建（轨道B） ───
  图谱同步 | 图谱探索 | 术语库 | 分类体系
─── 知识检索 ───
  向量索引 | RAG模拟
─── 配置 ───
  工作台配置 | 认知引擎配置
```

左侧sidebar加分组标签（`<div className="text-xs text-slate-500 px-2 py-1">知识准备（轨道A）</div>`），Tab按钮缩进显示。

## Risks And Compatibility

1. **6个无后端端点**：`/api/ontology/mappings`、`/api/ontology/export`、`/api/integration/*`、`/api/lineage/*`前端在用但无Java后端。knowledgeApi.ts中封装这些路径，但加fallback空响应，不阻塞前端。
2. **KnowledgeApiController占位**：index-status/sync/query三个端点返回占位数据。P3-2只扩展/rag，不改动已有端点签名。
3. **GlossaryController路径不匹配**：后端是`/api/glossary`，指令期望`/api/knowledge/glossary`。knowledgeApi.ts使用实际后端路径`/api/glossary`，不改后端路径。
4. **GraphCanvas已存在**：527行，接口与GraphExplorerTab的GraphNode/GraphEdge不完全对齐（Canvas用`Node`/`Link`，Explorer用`GraphNode`/`GraphEdge`）。拆分时需写适配层。
5. **AIPKnowledgeView.tsx重复**：239行，路由在`/knowledge_graph`，功能与RagTab重叠。本指令不清理，后续独立任务。
