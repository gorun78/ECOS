# Fix Ontology Workbench & Monitoring Center — Quality Remediation

## Goal

Fix all critical and high-severity gaps discovered in the audit of the ontology workbench Phase 2 and monitoring center 6-engine implementations. These tasks were done superficially — wiring was incomplete, i18n keys missing, API calls broken, and data flows inconsistent.

## Scope

- Fix **ontology workbench**: 9 critical/high issues + 5 medium issues
- Fix **monitoring center**: 4 high issues + 4 medium issues
- Non-goals: pre-existing TS errors (xyflow, api.ts), V6 ITaskAwareEngine, CognitiveEngineView settings (separate task)

## Part A: Ontology Workbench Fixes

### A1. Critical API Bugs

| # | Issue | File | Fix |

| 1 | `createRelationship()` argument order swapped — `(ontologyId, dto)` should be `(dto, ontologyId)` | [OntologyWorkbenchLayout.tsx](ecos_frontend/src/pages/OntologyWorkbenchLayout.tsx) L160 | Swap argument order to `(data, DEFAULT_ONTOLOGY_ID)` |

### A3. ObjectExplorerView Dual Data Source

**Problem:** Main table uses `fetchOntologyData()` API, but relations/instance counts/action-execution still use `datasets[].sampleData` (mock flow). Data is inconsistent.

**Fix approach:**
1. Remove `getInstancesOfObjectType()` that reads from `datasets[].sampleData`
2. Replace left sidebar instance counts (`datasets.find()` + `sampleData.length`) with API-sourced counts from `instanceData` or `PaginatedDataResponse.total`
3. Remove action execution writes to `sampleData` — either disable inline action execution or wire it to a proper API call
4. Fix `selectedInstance._index` usage — API-sourced instances don't have `_index`

### A4. MappingTab Save Sends Wrong Data

**Problem:** `handleSaveMapping` converts `propertyMappings` (Record<string,string>) to `fieldMappings` (Array<{source,target}>) but never sends `propertyMappings` itself. The backend `UpdateMappingDTO` may not understand `fieldMappings`.

**Fix:**
1. Add `propertyMappings?: Record<string,string>` to `UpdateMappingDTO` type
2. In `handleSaveMapping`, send both `fieldMappings` and `propertyMappings` on update
3. Verify dirty tracking resets correctly after save

## Part B: Monitoring Center Fixes

### B1. "Refresh All" Button Does Nothing

**Problem:** `refreshKey` state is incremented but never passed to `EngineCard`. Cards don't re-fetch.

**Fix:** Pass `refreshKey` as a `key` prop to each `EngineCard` so that changing `refreshKey` forces remount and re-fetch.

### B2. AiEngineAliasController Design

**Problem:** Two controllers serve the same `AiEngineImpl` at different paths (`/api/v1/engine/cognitive` and `/api/v1/engine/ai`), returning different health response shapes.

**Fix:** Unify health response shape. Make `AiEngineAliasController.health()` return `ApiResponse<HealthCheck>` (same as `AiEngineStatusController`), not `ApiResponse<Map<String, Object>>`. Or better: have `AiEngineAliasController` extend/delegate to the same health service method so responses are identical.

Also fix `AiEngineImpl.getName()` to return `"ai-engine"` (currently returns `"cognitive-engine"`) — or keep it as-is and have the alias controller use `engine.getName()` instead of hardcoding.

### B3. Health Response Shape Inconsistency

**Problem:** Security/data/ontology/ai controllers return `ApiResponse<HealthCheck>` with `{status, components}` nested structure. KB engine returns flat `ApiResponse<Map<String, Object>>` with `{status, engine, name, engineStatus, ...components}`. Frontend `HealthSummary` interface doesn't cleanly handle both.

**Fix:**
1. Make `KbEngineHealthController.health()` return `ApiResponse<HealthCheck>` (same as other controllers) instead of manually constructing a Map
2. Update `HealthSummary` interface in MonitoringCenter to include `components?: Record<string, any>` (already has it) — no change needed

### B4. EngineMonitor Double-Fetch & Raw Data Panel Label

**Problem:** EngineCard fetches health+status on mount. When expanded, EngineMonitor fetches the same health+config+status again. Also, raw data panel label says `/config` but for data engine the actual path is `/settings`.

**Fix:**
1. In EngineMonitor, accept optional `initialHealth` and `initialStatus` props from EngineCard to avoid re-fetching
2. Fix raw data panel label: use `meta.configPath || '/config'` for display

### B5. Tailwind Dynamic Class Issues

**Problem:** `hover:${styles.appBg}` and `${styles.cardBg}/50` use dynamic interpolation that Tailwind JIT can't compile.

**Fix:** Replace with static conditional classes or use inline styles for dynamic theming.

## Risks And Compatibility

- **Backend KbEngineHealthController change** — Changing from Map to HealthCheck may break other callers of `/api/v1/engine/knowledge/health`. Need to check for consumers.
- **ObjectExplorerView refactor** — Removing mock data flow may break action execution feature entirely. May need to stub out or disable action execution rather than wire to non-existent API.
- **Proposal changeType alignment** — Changing from NEW_OBJECT/NEW_LINK/MODIFY to CREATE/UPDATE/DELETE may affect the backend if it expects the former.
