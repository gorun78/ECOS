## 七、核心问题总结

### 7.1 架构问题

| # | 问题 | 严重度 | 影响范围 |
|---|------|--------|---------|
| 1 | **`ecos` package 混杂 8 个功能域** | 🔴 严重 | 60 个文件，占总量 31%，无内部分层 |
| 2 | **FrontendBridgeController 单类承载 6 个域** | 🔴 严重 | 657 行，无分层，半数端点含硬编码数据 |
| 3 | **Marketplace/Glossary 缺少 Service 层** | 🟡 中等 | Controller 直接操作 Repository |
| 4 | **Object-Runtime 的 Service 与 JdbcTemplate 混用** | 🟡 中等 | 5 个 Controller 模式不一致 |
| 5 | **PolicyEngineController 是空 Stub** | 🟡 中等 | 46 行占位代码，无实际实现 |
| 6 | **MenuController 直接查表无分层** | 🟡 中等 | 直接 JdbcTemplate 查 `TB_MENU_MODULE` |
| 7 | **ClassificationController 绕过 Compliance Service** | 🟡 中等 | 直接调用 HermesEngine |
| 8 | **`datanet` package 为空目录** | 🟢 低 | 无代码，应清理 |
| 9 | **Controller package 无子包划分** | 🟡 中等 | 42 个 Controller 平铺 |

### 7.2 两种代码风格并存

项目中同时存在两种代码组织风格：

| 风格 | 代表 Package | 特征 |
|------|-------------|------|
| **老风格**（接口-实现分离 + DAO 模式） | `iam`, `abac`, `audit`, `config`, `compliance`, `datapermission` | API 模块定义接口(`I*Service`, `*Dao`)，impl 模块实现(`*ServiceImpl`, `*DaoImpl`)，`model`/`entity` 独立 |
| **新风格**（Service + Repository + Entity 平铺） | `ecos` | 无接口抽象，Service/Repository/Entity/Event 全部平铺在同一 package，Repository 使用 JdbcTemplate（非 JPA） |

### 7.3 建议重构方向

1. **拆分 `ecos` package** — 按功能域分为 `ontology/`, `workflow/`, `agent/`, `dq/`, `marketplace/`, `glossary/`, `worldmodel/`, `objectruntime/`，每个子包内部按 `entity/`, `service/`, `repository/`, `event/` 分层
2. **拆分 FrontendBridgeController** — 每个功能域的端点迁移到对应 Controller 或独立 Service
3. **补齐缺失的 Service 层** — Marketplace、Glossary 补充 Service
4. **清理空 package** — 删除 `datanet/`
5. **统一代码风格** — 决策走「新风格」还是「老风格」，全项目统一
6. **Controller 分子包** — `/controller/ontology/`, `/controller/workflow/` 等
