## 1. 数据存储架构设计

### 1.1 DIKW 四层模型总览

ECOS 的核心理念是将企业数据从原始「数据」逐层加工为「智慧」，形成 DIKW 金字塔。每层对应不同的存储技术、数据形态和消费场景。

```
┌─────────────────────────────────────────────────────────────────┐
│  WISDOM    │ 决策智慧  │ PG (sys_man)          │ Goal, CausalLink │
│            │           │                       │ Scenario         │
├─────────────────────────────────────────────────────────────────┤
│ KNOWLEDGE  │ 知识      │ PG + pgvector +       │ Ontology, Rule   │
│            │           │ Neo4j (P1)            │ KnowledgeNode    │
├─────────────────────────────────────────────────────────────────┤
│INFORMATION │ 结构化信息│ PG (sys_man)          │ Object, Entity   │
│            │           │                       │ Metric, Schema   │
├─────────────────────────────────────────────────────────────────┤
│   DATA     │ 原始数据  │ PG + MinIO/Iceberg(P2)│ 业务表, 日志     │
│            │           │ + 外部数据源接入      │ 传感器, 文件     │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 各层详细设计

#### 1.2.1 Data 层（原始数据）

| 项目 | 内容 |
|------|------|
| **定位** | 企业原始业务数据的存储与管理，不做语义加工 |
| **存储** | PostgreSQL（当前）；P2 阶段引入 MinIO + Apache Iceberg 构建数据湖 |
| **数据内容** | 业务表（从外部系统接入/同步）、系统日志、用户上传文件、IoT 传感器数据、操作审计日志 |
| **接入方式** | datanet 模块的 DataSource/Connector 体系（已有 MySQL/PG/Oracle/ClickHouse 适配器） |

**技术决策分析**：

- **当前阶段 (MVP/P0)**：PostgreSQL 完全够用。所有业务数据通过 datanet 模块接入，存储在 PostgreSQL 的关系表或外部数据源（通过 federated query 或 ETL 同步）。
- **P2 阶段（Lakehouse）**：当数据量达到 TB 级或需要处理大量非结构化文件时，引入 MinIO（对象存储）+ Apache Iceberg（表格式），构建湖仓一体架构。参考 MVP v3 P2-1 方案：MinIO Docker 部署 → Iceberg Catalog 配置 → Pipeline 写入 → Trino/Spark SQL 查询。

**P2 湖仓架构**：
```
外部数据源 → Pipeline ETL → Iceberg表 (MinIO存储)
                                ↓
                         Trino/Spark SQL → ObjectQL 统一查询
```

#### 1.2.2 Information 层（结构化信息）

| 项目 | 内容 |
|------|------|
| **定位** | 经过清洗、建模、关联的结构化业务对象 |
| **存储** | PostgreSQL (sys_man 数据库) |
| **数据内容** | 业务对象（Customer/Supplier/Order）、实体定义、Schema 元数据、数据分类/分级、数据血缘、数据质量指标 |
| **核心能力** | Object Runtime 提供对象的完整 CRUD、状态机、Timeline、关系图谱 |

**当前表（已有）**：
- `ecos_ontology_entity` — 实体定义
- `ecos_ontology_property` — 属性定义
- `ecos_ontology_relationship` — 关系定义
- `ecos_ontology_action` — 动作定义
- 对象实例表 — 动态创建（由 Ontology 驱动的业务表）

**技术决策**：Information 层不需要独立存储引擎。PostgreSQL 的关系模型天然适合结构化对象的存储，JdbcTemplate 提供灵活的查询能力。在 ObjectQL 统一查询层（P1-6）实现后，上层应用通过 ObjectQL 访问，底层可透明切换存储。

#### 1.2.3 Knowledge 层（知识）

| 项目 | 内容 |
|------|------|
| **定位** | 从 Information 中提炼的业务规则、本体定义、知识图谱、语义关联 |
| **存储** | PostgreSQL JSONB（当前） + pgvector 向量索引（当前） + Neo4j 图数据库（P1-2） |
| **数据内容** | Ontology 本体定义、业务规则（Rule）、知识节点与关系、Agent 知识库、术语表（Glossary）、文档向量嵌入 |

**当前实现**：

| 能力 | 实现方式 | 成熟度 |
|------|---------|--------|
| 本体存储 | PG `ecos_ontology_*` 表 (JSONB) | ✅ 完整 |
| 向量检索 | pgvector + BGE-small-zh-v1.5 (512维) | ✅ 已集成 |
| 知识图谱 | PG JSONB 粗存 KnowledgeNode/KnowledgeEdge | ⚠️ 基础可用 |
| 术语表 | PG `ecos_glossary_*` 表 | ⚠️ 基础 CRUD |

**P1 阶段引入 Neo4j**（参考 MVP v3 P1-2）：

当前 PG JSONB 存储知识图谱节点和边的方式有两大限制：
1. **图遍历性能差**：递归 CTE 在深度超过 3-4 层时性能急剧下降
2. **图算法支持弱**：无法高效执行最短路径、中心度分析、社区发现等图算法

Neo4j 引入方案：
```
PG (Ontology 主数据) ←→ Neo4j (知识图谱)
        │                      │
        └──── 同步 ────────────┘
        │                      │
    ObjectQL 查询          Graph Explorer 可视化
                              Graph RAG 增强
```

**pgvector 是否足够？**  
当前场景（512 维向量，知识库文档量 < 10万篇）下 pgvector 完全够用：
- pgvector IVFFlat 索引在百万级向量下可达到 <10ms 的查询延迟
- 无需额外部署运维 Milvus/Qdrant 等专用向量数据库
- 数据与元数据在同一数据库，事务一致性有保障

**Milvus 引入时机**：当知识库文档量超过 500 万篇，或需要混合查询（标量过滤 + 向量检索）且 pgvector 性能不足时再评估。

#### 1.2.4 Wisdom 层（决策智慧）

| 项目 | 内容 |
|------|------|
| **定位** | 基于知识推理产生的决策方案、因果分析、情景模拟 |
| **存储** | PostgreSQL (sys_man) |
| **数据内容** | 企业目标（Goal）、因果链路（CausalLink）、情景模拟（Scenario）、What-If 分析结果、Agent 建议决策 |

**当前状态**（参考 S4-后端现状审计）：
- `WorldModelController` / `WorldModelService` / `WorldModelRepository` — 框架已存在
- `GoalEntity`, `CausalLinkEntity`, `ScenarioEntity` — Entity 已定义
- 实际业务逻辑待 P0-4 补齐

**存储选型**：Wisdom 层数据量小（目标数 < 1000，因果链 < 10000，情景 < 1000），不需要专用存储。PostgreSQL 的 ACID 事务、JSONB 灵活字段、外键约束完全满足需求。

### 1.3 是否需要引入 OLAP 引擎（Doris / ClickHouse）？

**短期（P0-P1）**：不需要。当前场景以 OLTP 为主（对象 CRUD + 工作流 + Agent 交互），分析类查询通过 PostgreSQL 的聚合查询即可满足。

**中期（P2+）**：当出现以下场景时引入 ClickHouse 或 Apache Doris：
- 经营看板需要秒级响应 TB 级历史数据的多维聚合（如按月/季/年统计供应商质量趋势）
- Pipeline 节点产生大量时序指标数据（数据质量检查结果、Agent 执行指标）
- 实时监控需要亚秒级聚合查询

**推荐选型**：ClickHouse → 单表查询性能极致，适合时序指标分析；Doris → 多表 JOIN 能力更强，适合多维分析看板。ECOS 的监控和看板场景偏单表/星型模型聚合，ClickHouse 更合适。

### 1.4 存储架构最终推荐

```
┌──────────────────────────────────────────────────────────────────┐
│  阶段       │ 存储组件               │ 用途                       │
├──────────────────────────────────────────────────────────────────┤
│  P0 (当前)  │ PostgreSQL (sys_man)   │ 全部四层 (DIKW)            │
│            │ pgvector               │ 向量检索 (Knowledge 层)     │
│            │ PostgreSQL (ecos_know) │ 知识库独立库                │
├──────────────────────────────────────────────────────────────────┤
│  P1 (3-6月) │ + Neo4j               │ 知识图谱 (Knowledge 层)     │
│            │ + 统一查询层(ObjectQL) │ 屏蔽底层存储差异             │
├──────────────────────────────────────────────────────────────────┤
│  P2 (6-12月)│ + MinIO + Iceberg     │ 数据湖 (Data 层)            │
│            │ + ClickHouse           │ 时序/OLAP (Information 层)  │
│            │ + Kafka                │ 事件总线 (跨层)             │
└──────────────────────────────────────────────────────────────────┘
```

## 2. 数据流设计

### 2.1 完整数据流架构

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│  数据源   │ → │  接入层   │ → │  处理层   │ → │  存储层   │ → │  服务层   │ → │  消费层   │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
   业务系统       datanet        buszhi          DIKW           API           前端/Agent
   IoT/日志       Connector      Pipeline        PG/MinIO       REST/QL       应用/看板
   文件/API       数据源管理      Workflow        Neo4j          WebSocket     外部系统
                               Agent/DQ         ClickHouse     Agent API
```

### 2.2 各层详细说明

#### 2.2.1 数据源层

| 数据源类型 | 示例 | 接入方式 |
|-----------|------|---------|
| 关系型数据库 | MySQL, Oracle, SQL Server, PostgreSQL | JDBC Connector (已有) |
| API 数据源 | REST API, GraphQL | HTTP Connector |
| 文件数据源 | CSV, Excel, JSON, Parquet | File Connector + MinIO (P2) |
| 流式数据源 | Kafka, MQTT | 消息 Connector (P2) |
| SaaS 数据源 | ERP, CRM, HRM | API Connector |
| Agent 产出 | LLM 分析结果, 自动分类/分级 | 直接写入 sys_man |
| 用户输入 | c2eos 前端表单, 文件上传 | REST API → sys_man |

#### 2.2.2 接入层（datanet 模块）

**当前状态**：已有 `DataSourceService`、`CatalogService`、`MetadataService`，支持 MySQL/PG/Oracle/ClickHouse 四类数据源的注册、连接测试、元数据扫描。

**数据格式**：
- 结构化数据 → `DataSourceDescriptor` DTO（描述数据源连接信息）
- 元数据 → `SchemaRegistryService` 注册表结构
- 血缘信息 → `LineageRecorder` / `LineageEvent`

**关键流程**：
```
注册数据源 → 连接测试 → 元数据扫描 → Schema 注册 → 数据目录发布
```

#### 2.2.3 处理层（buszhi 模块 + runtime）

**当前状态**：
- `runtime-core/transform.*` — 数据转换管道（DataCleansingStep, FieldMappingStep 等）
- `runtime-core/quality.*` — 数据质量规则执行
- `WorkflowEngine` + `Pipeline`（已从 Workflow 分离为独立实体）
- `AgentRuntime` — ReAct 引擎 + 工具调用 + 推理

**数据格式转换**：
```
原始数据 → 清洗 (CLEANSE) → 转换 (TRANSFORM) → 校验 (VALIDATE) → 充实 (ENRICH)
                │                  │                  │                │
           DataCleansingStep   FieldMappingStep   QualityRule     Agent/AI
           FormatConverter      Schema映射          DqService      Ontology
```

**处理模式**：

| 模式 | 引擎 | 场景 | 延迟 |
|------|------|------|------|
| 批处理 | Pipeline + TaskScheduler | 每日 ETL、数据质量巡检 | 分钟级 |
| 事件驱动 | Spring EventBus + Workflow | 对象创建/更新→触发流程 | 秒级 |
| Agent 智能 | AgentRuntime (ReAct) | 语义理解、自动分级、知识抽取 | 秒~分钟级 |
| 实时流 | Kafka (P2) | 传感器/日志实时处理 | 毫秒级 |

#### 2.2.4 存储层

已在第 1 章详述。数据流视角的存储路径：

```
处理完成的结构化对象 → PostgreSQL sys_man (DIKW 四层表)
处理完成的文档知识 → PostgreSQL ecos_knowledge + pgvector embedding
Pipeline 产出的数据 → Data 层表 / Iceberg (P2)
Agent 产出的知识 → Knowledge 层 (PG JSONB / Neo4j)
系统审计日志 → ecos_audit_log 表
```

#### 2.2.5 服务层

| 服务 | 暴露方式 | 功能 |
|------|---------|------|
| Object Runtime API | REST `/api/v1/ecos/objects/*` | 对象 CRUD、状态机、关系图、Timeline |
| Ontology API | REST `/api/v1/ecos/ontologies/*` | 本体建模、实体/属性/关系管理 |
| Workflow API | REST `/api/v1/ecos/workflows/*` | 工作流定义/实例/任务/审批 |
| Agent API | REST `/api/agent/*` | Agent 配置/调用/Mesh/Profile |
| ObjectQL | REST `/api/query/objectql` | 统一对象查询语言 (P1-6) |
| Knowledge Search | REST `/api/knowledge/search` | 语义搜索 + 向量检索 |
| Data Catalog | REST `/api/catalog/*` | 数据资产浏览、分类分级 |
| 经营看板 | REST `/api/v1/ecos/biz/dashboard` | 聚合指标 API |
| WebSocket | WS `/ws/agent/*` | Agent 执行实时 trace 推送 (P3) |
| Gateway | Spring Cloud Gateway | 统一入口、鉴权、限流、路由 |

#### 2.2.6 消费层

| 消费者 | 消费方式 | 场景 |
|--------|---------|------|
| c2eos 前端 | REST API / WebSocket | 11 页面（AgentStudio, DataCatalog, COG, Monitoring 等）|
| Agent Runtime | 内部 API 调用 | Agent 调用工具 → object_query, ontology_explore 等 |
| 外部系统 | REST API / Webhook | ERP/CRM 读取 ECOS 数据，或 ECOS 推送结果 |
| Low-Code Builder (P2) | REST API | 拖拽搭建应用，底层调用 ObjectQL |
| 统一门户 (P2) | REST API | 工作台首页、全局搜索、待办通知 |

### 2.3 端到端数据流示例：供应商准入审查

```
1. 用户上传供应商资料 (c2eos 前端)
        ↓ POST /api/v1/ecos/objects/Supplier
2. Object Runtime 创建 Supplier 对象
        ↓ 发布 ObjectCreatedEvent
3. Workflow 引擎自动触发「供应商准入审查流程」
        ↓ 流程节点: Agent Task
4. Agent Runtime 执行风险评估:
   - object_query → 查供应商档案 (Information 层)
   - ontology_explore → 查关联关系 (Knowledge 层)
   - knowledge_search → 查准入标准文档 (Knowledge 层/向量检索)
        ↓ 
5. Agent 输出风险评估报告 → 写入 DqIssue (质量层)
        ↓ 流程节点: User Task
6. 审批人查看报告 → 决策通过/驳回
        ↓ 通过: transition to "Approved"
7. 供应商状态变更 → 自动更新经营看板指标
```

## 3. 数据处理引擎选型

### 3.1 候选引擎对比

| 维度 | Apache Spark | Apache Flink | Apache Doris | PostgreSQL |
|------|-------------|-------------|-------------|------------|
| **核心定位** | 批处理引擎 | 流处理引擎 | OLAP 分析引擎 | OLTP 事务数据库 |
| **处理模型** | 微批 (Structured Streaming 也支持流) | 原生流处理 (事件级别) | MPP 列存查询 | 行存事务处理 |
| **延迟** | 秒~分钟级 | 毫秒~秒级 | 毫秒~秒级 | 毫秒级 (OLTP) |
| **吞吐** | 极高 (TB/PB 级) | 极高 (百万事件/秒) | 高 (亿行/秒扫描) | 中等 (万 QPS) |
| **事务支持** | ❌ 不支持 | ❌ 不支持 | ❌ 弱事务 | ✅ 完整 ACID |
| **SQL 兼容** | Spark SQL (ANSI SQL 子集) | Flink SQL | MySQL 协议兼容 | PostgreSQL 方言 |
| **运维复杂度** | 高 (需 YARN/K8s + HDFS/S3) | 高 (需 ZK + 状态后端) | 中 (FE+BE 集群) | 低 (单机/主从) |
| **Java 集成** | ✅ 原生 API | ✅ 原生 API | ✅ JDBC | ✅ JDBC/JdbcTemplate |
| **生态丰富度** | ⭐⭐⭐⭐⭐ (MLlib, GraphX) | ⭐⭐⭐⭐ (CDC, Kafka) | ⭐⭐⭐ (BI 对接) | ⭐⭐⭐⭐⭐ (扩展/插件) |

### 3.2 ECOS 场景适用性分析

#### 场景一：大规模 ETL / 数据批处理

| 场景 | 推荐引擎 | 理由 |
|------|---------|------|
| 每日从 100+ 数据源同步全量数据 | **Spark** (P2) | 天然适合大规模批处理，支持多数据源并行读取 |
| 历史数据重处理/回刷 | **Spark** | 分布式计算，TB 级数据可在分钟级完成 |
| 小规模定时同步 (<10 数据源) | **PostgreSQL + Pipeline** | 当前 Pipeline 已就绪，无需引入重量级引擎 |

#### 场景二：实时事件处理 / 流式管道

| 场景 | 推荐引擎 | 理由 |
|------|---------|------|
| IoT 传感器数据实时处理（未来） | **Flink** (P3) | 毫秒级延迟，支持 CEP 复杂事件处理 |
| 对象变更事件 → 实时触发工作流 | **Spring EventBus** (P0) → **Kafka** (P2) | MVP 阶段 Spring Event 已够用，P2 引入 Kafka 解耦 |
| Agent 执行实时 trace 推送 | **WebSocket** (P3) | 前端实时展示，无需流处理引擎 |

#### 场景三：多维分析 / OLAP 查询

| 场景 | 推荐引擎 | 理由 |
|------|---------|------|
| 经营看板（月度/季度聚合） | **PostgreSQL** (P0/P1) | 数据量小（万级），PG 聚合查询已够用 |
| 数据质量趋势分析（TB 级历史） | **ClickHouse** (P2) | 列存 + 向量化执行，聚合查询比 PG 快 100x |
| 多维交叉分析（供应商×品类×时间） | **Apache Doris** (P2) | JOIN 能力强，MySQL 兼容，BI 工具直接对接 |

#### 场景四：OLTP 事务处理（核心场景）

| 场景 | 推荐引擎 | 理由 |
|------|---------|------|
| 对象 CRUD + 状态机 + 关系管理 | **PostgreSQL** | ✅ 当前唯一选择，ACID 保证数据一致性 |
| 工作流实例状态管理 | **PostgreSQL** | 工作流状态机强依赖事务 |
| Agent 配置/会话/记忆 | **PostgreSQL** | 小数据量，事务一致性关键 |

### 3.3 最终推荐组合

```
┌─────────────────────────────────────────────────────────────────────┐
│  阶段     │ 批处理 ETL    │ 流处理         │ OLAP 分析     │ OLTP     │
├─────────────────────────────────────────────────────────────────────┤
│ P0 (当前) │ PG + Pipeline │ Spring EventBus │ PG 聚合查询   │ PG       │
│           │ (轻量级)      │ (进程内)        │ (小数据量)    │ (核心)   │
├─────────────────────────────────────────────────────────────────────┤
│ P1        │ PG + Pipeline │ Spring EventBus │ PG + 物化视图 │ PG       │
│           │               │                 │ + ObjectQL    │          │
├─────────────────────────────────────────────────────────────────────┤
│ P2        │ + Spark       │ + Kafka         │ + ClickHouse  │ PG       │
│           │ (批处理加速)  │ (事件解耦)      │ / Doris       │          │
├─────────────────────────────────────────────────────────────────────┤
│ P3        │ Spark         │ + Flink         │ ClickHouse    │ PG       │
│           │               │ (实时流)        │               │          │
└─────────────────────────────────────────────────────────────────────┘
```

**核心原则**：

1. **PostgreSQL 是唯一不可替代的引擎**：作为 DIKW 四层的主存储和 OLTP 事务引擎，承担所有写操作和核心读操作。
2. **先做减法，再做加法**：P0 阶段用 PG + Pipeline + Spring EventBus 覆盖所有场景，验证业务闭环后再引入专用引擎。
3. **专用引擎按场景引入**：不是「技术驱动」而是「场景驱动」——当 PG 的某项能力确实成为瓶颈时，才引入专用引擎。
4. **统一查询层抽象**：ObjectQL 屏蔽底层存储差异，上层应用不感知底层是 PG/ClickHouse/Spark。

## 5. 搜索方案设计

### 5.1 候选方案对比

| 维度 | PostgreSQL 全文搜索 | Elasticsearch | pgvector 向量搜索 |
|------|-------------------|---------------|-------------------|
| **核心原理** | tsvector + GIN 索引 | 倒排索引 + BM25 | 向量相似度 (cosine/L2) |
| **最佳场景** | 精确关键词匹配 | 全文检索 + 聚合 | 语义相似度搜索 |
| **部署复杂度** | 零（PG 内置） | 高（独立集群） | 低（PG 扩展） |
| **中文支持** | zhparser / simple | IK 分词器 | BGE 模型语义理解 |
| **同义词/模糊** | ⚠️ 需手动配置 | ✅ 内置支持 | ✅ 语义自然覆盖 |
| **结构化过滤** | ✅ 原生 SQL | ✅ Bool Query | ⚠️ 需预过滤 + 后过滤 |
| **规模上限** | 百万级 | 十亿级 | 百万级 (pgvector IVFFlat) |
| **运维成本** | 零 | 高（JVM 调优、集群管理） | 低（PG 管理） |
| **当前集成** | ❌ 未使用 | ❌ 未部署 | ✅ BGE-small-zh-v1.5, 512维 |

### 5.2 ECOS 搜索场景分析

ECOS 涉及 4 类主要搜索场景：

| 场景 | 搜索内容 | 用户意图 | 最佳技术 |
|------|---------|---------|---------|
| **语义知识搜索** | 知识库文档、准入标准、合规政策 | 「供应商准入需要什么条件」 | **pgvector 向量搜索** (已有) |
| **对象精确搜索** | 供应商名称、合同编号、客户代码 | 「星辰科技」「PO-2025-001」 | **PG 全文搜索 (tsvector)** 或 LIKE |
| **跨模块全局搜索** | 混合搜索对象、Agent、Workflow、术语 | 「审批流程相关的所有内容」 | **搜索网关 + 分发 + 聚合** |
| **自然语言查询** | 「华东地区高价值客户」 | 语义理解→ObjectQL | **Agent NLQ + 语义翻译** (P1-6) |


### 5.4 搜索引擎分阶段引入策略

#### P0 阶段（当前-2个月）：PG 双索引

```
PostgreSQL (sys_man)
├── pgvector IVFFlat 索引 → 语义搜索 (BGE 512维)
└── GIN 索引 (tsvector)  → 关键词搜索 (对象名称/编号)
```

**关键实现**：
- 对象表添加 `search_vector tsvector` 列，GIN 索引
- 中文分词使用 `zhparser` 扩展或 `simple` 词典 + ICU 分词
- 知识库搜索全部走 pgvector 向量检索（当前唯一实现）

#### P1 阶段（3-6个月）：搜索网关 + 混合检索

- 实现 Search Gateway Controller：`POST /api/search`
- 混合检索策略：BM25 关键词分数 + 向量相似度分数 → 加权融合
- 跨模块聚合：同一搜索词分发到对象索引、知识库、Agent、Workflow，聚合展示
- 搜索历史记录 → 反馈优化排序

#### P2 阶段（6-12个月）：可选引入 Elasticsearch

**Elasticsearch 引入条件**（满足任一）：
- 知识库文档量超过 100 万篇
- 需要复杂的分词/同义词/拼音搜索
- 需要实时日志搜索（审计日志 > 1000 万条）
- 需要聚合分析（搜索热词、用户搜索行为分析）

**引入方式**：作为 PG 搜索的补充，而非替代。PG 仍然是主存储，ES 通过 CDC (Kafka) 同步索引数据。

### 5.6 最终推荐结论

| 阶段 | 方案 | 覆盖场景 |
|------|------|---------|
| **P0** | PostgreSQL (pgvector + tsvector) | 语义搜索 + 关键词搜索，覆盖全部 4 类场景 |
| **P1** | + 搜索网关 + 混合检索 | 统一入口、跨模块聚合、加权融合排序 |
| **P2** | + Elasticsearch (按需) | 大规模全文检索、搜索分析、日志搜索 |

**核心理由**：不需要 Elasticsearch 起步。ECOS 当前搜索场景规模可控（百万级文档、十万级对象），PostgreSQL 的 pgvector（语义） + tsvector（关键词）组合已经覆盖全部需求。搜索网关在 P1 阶段实现的投入产出比最高——它能统一搜索体验，而底层索引引擎可以后续平滑升级为 ES 而无需改变 API。
