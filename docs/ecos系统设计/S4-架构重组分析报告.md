## 六、依赖关系分析

### 6.1 当前依赖链

```
databridge-common (无依赖)
     ↑
databridge-runtime (依赖 common)
     ↑
databridge-sysman (依赖 runtime + common)
databridge-datanet (依赖 common)
databridge-buszhi (空壳)
databridge-dccheng (空壳)
databridge-aimod (空壳)
```

### 6.2 重组后依赖链

```
databridge-common
     ↑
databridge-runtime (精简为: task + monitor + crypto + dataaccess + config + logging)
     ↑
├── databridge-sysman (IAM + ABAC + 审计 + 配置)
├── databridge-datanet (数据接入)  ← 依赖 common 即可
├── databridge-buszhi (数据处理+质量+流水线)
│     └── 依赖 databridge-dccheng (通过 PipelineEvent 解耦)
├── databridge-dccheng (术语+本体+知识图谱)
├── databridge-aimod (Agent Mesh + LLM)
├── databridge-portal (门户)
├── databridge-market (市场)
├── databridge-worldmodel (世界模型)
└── databridge-workspace (工作区)
```

### 6.3 关键解耦点

| 耦合点 | 当前状态 | 建议 |
|--------|---------|------|
| sysman-impl 内部跨域引用 | Ontology ↔ Workflow ↔ Dq 直接 import | 拆分到不同模块后自然解耦 |
| ObjectRuntimeEvent | sysman-impl 中定义 | 迁移到 buszhi 或 common 的 event 包 |
| PipelineEvent (common) | Bus-Zhi ↔ Dc-Cheng 已通过事件解耦 | ✅ 设计正确，保持 |
| ExpressionService | 被 Workflow 和 Ontology Rule 共用 | 迁到 common 或保留在 buszhi 作为 SPI |

### 阶段 0: 确认空壳模块 (立即)

## 九、总结

### 核心发现

1. **sysman-impl 严重超载**: 35 个 Controller 实际涵盖了至少 8 个不同功能域，是本次重组的核心目标。
2. **buszhi/dccheng/aimod 空壳**: 这 3 个模块已声明但无代码，正好作为迁移目标。
3. **runtime-core 过度庞大**: 300+ 个类混在一起，包含 Agent Mesh、数据质量评估、数据转换管道等不同关注点。
4. **datanet 结构良好**: 是唯一结构清晰、职责单一的模块，可作为重组参考模板。
5. **common 层设计合理**: 异常体系、事件机制、DTO 共享已按 ADR 规范执行。

### 重组收益

- **模块职责清晰**: 每个模块只负责一个功能域
- **独立部署**: 各模块可独立构建、测试、部署
- **团队并行**: 不同团队可并行开发不同模块
- **降低认知负载**: 新人只需理解相关模块即可
