# 问题1: 用户角色体系

## 1.1 当前状态

| 维度 | 现状 |
|------|------|
| **角色数量** | 仅 2 个正式角色: `ROLE_SUPER_ADMIN`（超级管理员）、`admin`（系统管理员） |
| **DB 种子数据** | `TD_ROLE` 表有 3 条: `R001`(admin)、`R002`(operator)、`R003`(user) — 仅 admin 被实际使用 |
| **用户数量** | 13 个用户（含 admin 种子用户） |
| **权限粒度** | 仅有基础 RBAC: user/role/organization/permission CRUD 的 READ/WRITE/DELETE 权限，共 20 条 permission seed |
| **前端页面** | UserManagement 页面支持 Users/Roles/Orgs/Permissions 四个 Tab |
| **核心问题** | ① 角色仅区分"管理员 vs 普通用户"，与实际企业层级完全不匹配；② 无部门/业务角色概念；③ 所有业务页面无角色感知，13 个用户看到相同界面 |

## 1.2 目标状态

建立 **7 种角色** 的分层角色体系，覆盖从系统运维到业务执行的全链路用户群体。每个角色有明确的：

- **职责边界**：该角色核心负责什么
- **可见模块**：在 Sidebar 导航和路由层面能看到的页面
- **典型使用场景**：该角色登录 ECOS 后的"一天工作流"

## 1.3 角色矩阵

### 角色总览表

| # | 角色名称 | 角色代码 | 用户画像 | 核心职责 | 用户占比 |
|---|---------|---------|---------|---------|:----:|
| 1 | **超级管理员** | `ROLE_SUPER_ADMIN` | 系统运维负责人 | 全系统配置、IAM 管理、安全策略、系统级操作 | ~5% |
| 2 | **企业高管** | `ROLE_EXECUTIVE` | CEO/CFO/CTO | 战略看板、经营分析、目标制定、审批决策 | ~5% |
| 3 | **部门管理员** | `ROLE_DEPT_ADMIN` | 部门经理/总监 | 部门 OKR 拆解、项目审批、团队绩效、部门数据 | ~15% |
| 4 | **数据管理员** | `ROLE_DATA_ADMIN` | 数据架构师/数据工程师 | 数据目录管理、Pipeline 编排、数据质量、Ontology 建模 | ~10% |
| 5 | **安全审计员** | `ROLE_AUDITOR` | 安全/合规/审计人员 | 安全审计查看、ABAC 策略审核、加密审计、合规检查 | ~5% |
| 6 | **业务操作员** | `ROLE_OPERATOR` | 业务执行人员 | 对象数据填报、流程发起、任务执行、数据查询 | ~40% |

### 详细角色说明

#### 角色1: 超级管理员 (`ROLE_SUPER_ADMIN`)

| 维度 | 内容 |
|------|------|
| **职责** | 系统级配置管理、IAM 全权限管理（用户/角色/机构/权限 CRUD）、安全策略配置、系统字典管理、全局参数配置 |
| **可见模块** | 🔓 全部页面（无限制） |
| **典型场景** | ① 创建新用户并分配角色和机构 → ② 配置 ABAC 安全策略 → ③ 设置系统密级和审计模式 → ④ 管理数据字典和系统配置 → ⑤ 查看全局审计日志 |
| **权限收束** | 拥有所有 `*:*` 通配权限，包括 `user:*` / `role:*` / `permission:*` / `organization:*` / `audit:*` / `config:*` |
| **安全约束** | 操作需强制审计记录；高危操作（删除用户/角色）需二次确认 |

#### 角色2: 企业高管 (`ROLE_EXECUTIVE`)

| 维度 | 内容 |
|------|------|
| **职责** | 制定企业战略目标、查看经营仪表盘、审批关键决策、What-If 情景推演分析 |
| **可见模块** | Mission Control(总控台)、BizDashboard(经营仪表盘)、WorldModelViewer(战略目标)、CognitiveOperatingSystem(认知总控)、MonitoringCenter(告警概览)、OperationsDashboard(产值看板)——均为只读 |
| **典型场景** | ① 登录查看 BizDashboard 营收/利润/回款 KPI → ② 打开 WorldModel 查看年度目标执行进度 → ③ 进行 What-If 情景推演（如"若合同额下降15% 对利润的影响"） → ④ 审批关键流程待办 |
| **权限收束** | `dashboard:read` / `goal:read` / `scenario:*` / `workflow:approve` / `report:read` — 仅可查看/审批，不可修改底层数据和配置 |

#### 角色3: 部门管理员 (`ROLE_DEPT_ADMIN`)

| 维度 | 内容 |
|------|------|
| **职责** | 拆解企业战略目标为本部门 OKR、管理本部门项目/合同/产值、审批部门内流程、查看部门绩效 |
| **可见模块** | WorldModelViewer(目标管理·本部门)、BizDashboard(部门维度)、ProjectTracker(项目跟踪)、ContractManager(合同管理)、OperationsDashboard(产值分配)、WorkflowDesigner(流程查看/审批)、KanbanBoard(项目看板)、OperationalApps(运营应用) |
| **典型场景** | ① 从企业战略目标拆解出部门 Q3 OKR → ② 创建部门项目并跟踪里程碑 → ③ 审批合同登记/计量/开票流程 → ④ 查看部门产值分配看板 → ⑤ 通过 Kanban 管理团队任务 |
| **权限收束** | 操作范围按 `org_id` 限定本部门及子部门；`goal:write` / `project:write` / `contract:write` / `workflow:approve` 限定 scope=本部门 |

#### 角色4: 数据管理员 (`ROLE_DATA_ADMIN`)

| 维度 | 内容 |
|------|------|
| **职责** | 数据资产目录管理、Pipeline 编排与调度、数据质量规则配置、Ontology 建模与语义层设计、知识图谱维护 |
| **可见模块** | DataCatalog(数据目录)、DatasetExplorer(数据集详览)、DataLineage(数据血缘)、PipelineBuilder(流水线编排)、CodeWorkbook(代码工作簿)、DataQualityDashboard(数据质量)、OntologyExplorer(本体浏览)、OntologyDesigner(本体设计器)、ObjectExplorer(数据浏览器)、GlossaryManager(术语库)、SystemDictionary(数据字典) |
| **典型场景** | ① 注册新数据源并创建数据资产 → ② 设计 Pipeline 从 Raw→Business 数据加工 → ③ 配置数据质量规则自动检测 → ④ 在 Ontology Designer 中定义新实体(如 Supplier) → ⑤ 查看数据血缘定位上游影响 |
| **权限收束** | `datasource:*` / `pipeline:*` / `ontology:*` / `quality:*` / `glossary:*` / `knowledge:*` — 有写权限但无 IAM 和系统配置权限 |

#### 角色5: 安全审计员 (`ROLE_AUDITOR`)

| 维度 | 内容 |
|------|------|
| **职责** | 查看安全审计日志、审核 ABAC 策略合规性、检查加密审计链、验证数据脱敏效果、生成合规报告 |
| **可见模块** | 安全中心(SecurityCenter 统一入口 → 含 Dashboard/SecurityAudit/CryptoAuditPanel)、AbacPolicyManager(只读审核)、DataMaskingDemo(验证脱敏)、UserManagement(只读用户/角色查看)、MonitoringCenter(安全告警) |
| **典型场景** | ① 打开安全中心 Dashboard 查看安全态势 → ② 查看近期安全审计事件 → ③ 审核新增 ABAC 策略是否合规 → ④ 验证数据脱敏规则的覆盖率和效果 → ⑤ 生成月度安全合规报告 |
| **权限收束** | `audit:*`(不含删除)、`abac:read`、`security:read` — 只读为主，不可修改安全策略和用户数据 |

#### 角色6: 业务操作员 (`ROLE_OPERATOR`)

| 维度 | 内容 |
|------|------|
| **职责** | 日常业务操作：数据填报(CURD)、流程发起与执行、任务处理、数据查询与导出 |
| **可见模块** | ObjectExplorer(对象浏览/创建/编辑)、WorkflowDesigner(发起流程/处理待办)、OperationalApps(运营工作台)、ProjectTracker(项目操作)、ContractManager(合同操作)、OperationsDashboard(产值操作)、AgentStudio(Agent 对话·受限)、AgentMesh(任务发起·受限)、BizDashboard(部门看板) |
| **典型场景** | ① 打开 ObjectExplorer 创建新客户记录 → ② 通过 OperationalApps 提交合同登记 → ③ 处理 Workflow 待审批 → ④ 在 AgentStudio 与 Agent 对话查询业务数据 → ⑤ 导出报表 |
| **权限收束** | `object:read|write|create`(不可 delete)、`workflow:start|execute`、`project:read|write`、`contract:read|write` — 可增改查但不可删，不可修改系统配置和 Ontology 定义 |

#### 角色7: 只读观察员 (`ROLE_VIEWER`)

| 维度 | 内容 |
|------|------|
| **职责** | 浏览数据资产、查看报表和仪表盘、导出数据（仅读，无任何写权限） |
| **可见模块** | DataCatalog(只读浏览)、DatasetExplorer(只读查看)、DataLineage(只读)、BizDashboard(只读)、KnowledgeGraph(只读)、MarketplaceBrowser(只读)、MonitoringCenter(只读概览) |
| **典型场景** | ① 浏览数据目录了解企业数据资产 → ② 查看数据集 Schema 和样本数据 → ③ 查看经营仪表盘 KPI → ④ 导出数据报表 |
| **权限收束** | 全局 `*:read` — 仅读权限，不可创建/修改/删除任何数据 |

# 问题2: 安全机制 UX 简化

## 2.1 当前状态

| 维度 | 现状 |
|------|------|
| **安全相关页面** | 6 个独立页面: `SecurityConfigPanel`、`SecurityAudit`、`CryptoAuditPanel`、`AbacPolicyManager`、`DataMaskingDemo`、`UserManagement` |
| **Sidebar 导航** | "安全与审计" 分组下列出 5 个平铺入口（不含 UserManagement），无层级关系 |
| **功能完备度** | ✅ 功能全面（ABAC 策略 CRUD、数据脱敏实时演示、安全配置管理、审计日志查看、加密审计链校验） |
| **核心问题** | ① **入口碎片化**：6 个页面散落在 Sidebar 中，用户不知道从哪里开始；② **无引导**：缺乏"安全态势总览→发现问题→解决问题"的工作流；③ **角色混淆**：管理员和审计员看到相同界面，但职责完全不同；④ **门槛过高**：ABAC 策略编辑器直接暴露底层 expression，业务用户无法使用 |

### 现有安全页面功能清单

| 页面 | 功能 | 目标用户 |
|------|------|---------|
| SecurityConfigPanel | 密级设置(1-5级)、工作站绑定、审计模式(基础/详细)、沙箱模式开关 | 超级管理员 |
| SecurityAudit | 审计日志列表、Top Actors/Actions 洞察、近期事件 | 审计员 |
| CryptoAuditPanel | 密码学级审计链验证、多链路 Trace 遥测签名 | 审计员 |
| AbacPolicyManager | ABAC 策略 CRUD、条件表达式编辑、分页搜索 | 超级管理员 |
| DataMaskingDemo | 脱敏规则选择(手机/邮箱/身份证等)、实时脱敏效果对比 | 审计员/数据管理员 |
| UserManagement | 用户/角色/组织机构/权限 CRUD | 超级管理员 |

## 2.2 目标 UX 架构: **安全中心 (Security Center)**

### 设计理念

> **"一个入口、三步引导、两类视图"**  
> 将所有安全相关功能收敛至统一的安全中心门户，按用户角色分流至不同视图，以 Dashboard→向导→深度操作三级递进降低认知负荷。

## 3.3 目标管理全生命周期

### 3.3.1 四阶段流程

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ ① 创建   │───→│ ② 拆解   │───→│ ③ 执行   │───→│ ④ 复盘   │
│  & 对齐  │    │  & 关联  │    │  追踪    │    │  & 优化  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
```

### 3.3.2 阶段详解

#### 阶段①: 创建 & 对齐 (Create & Align)

| 步骤 | 操作 | UI 交互 |
|:----:|------|---------|
| 1 | **创建战略目标** (L1) | EXECUTIVE 角色在 WorldModelViewer → "新建战略目标" → 填写名称/描述/量化目标值/时间范围 |
| 2 | **对齐确认** | 系统自动检测同级目标权重之和 = 100，子目标权重之和 = 父目标权重，冲突时高亮警告 |
| 3 | **发布** | 目标状态: `DRAFT` → `PUBLISHED`，发布后通知相关部门负责人 |

#### 阶段②: 拆解 & 关联 (Decompose & Link)

| 步骤 | 操作 | UI 交互 |
|:----:|------|---------|
| 1 | **一键拆解** | DEPT_ADMIN 进入战略目标详情 → 点击「拆解为部门 OKR」→ 系统自动创建子目标节点，预填父目标信息 |
| 2 | **手动拆解** | 在目标树中右键 → "添加子目标" → 选择类型 (OKR/KPI) → 填写表单 |
| 3 | **关联工作流** | 在 KPI 目标详情中 → "关联工作流" → 选择已有 Workflow 或跳转 WorkflowDesigner 新建 |
| 4 | **Agent 绑定** | 为工作流节点绑定 Agent 任务（如"自动生成周报"、"智能异常检测"） |

#### 阶段③: 执行追踪 (Execute & Track)

| 步骤 | 操作 | UI 交互 |
|:----:|------|---------|
| 1 | **实时看板** | WorldModelViewer 新增「执行追踪」Tab，显示目标树 + 进度条 + 红绿灯状态 |
| 2 | **自动更新** | 工作流完成节点 → 自动更新关联 KPI 的 `current_value`；Agent 完成分析 → 写入分析结果 |
| 3 | **手动更新** | 操作员可手动更新 `current_value` 并附注说明 |
| 4 | **偏差告警** | 当 `current_value / target_value * 时间进度` < 80% 时，自动标黄；< 50% 标红并通知负责人 |
| 5 | **趋势图** | 展示 `current_value` 随时间变化的折线图，对比计划曲线 vs 实际曲线 |

#### 阶段④: 复盘 & 优化 (Review & Optimize)

| 步骤 | 操作 | UI 交互 |
|:----:|------|---------|
| 1 | **周期复盘** | 目标到达 `end_date` 或手动触发复盘 → 自动生成复盘报告 |
| 2 | **复盘报告** | 目标达成率、偏差分析、关联工作流完成统计、Agent 贡献度、因果链影响分析 |
| 3 | **经验沉淀** | 成功策略标记为"最佳实践"，失败原因录入知识库，供未来 What-If 推演参考 |
| 4 | **目标归档** | 状态: `COMPLETED` / `ARCHIVED`，保留历史数据供趋势分析 |

### 3.4.3 与工作流/Agent 联动

| 联动场景 | 触发条件 | 自动化行为 |
|---------|---------|-----------|
| **目标→工作流** | KPI 目标创建时关联 Workflow | Workflow 执行进度反向更新 KPI 的 `current_value` |
| **工作流→Agent** | Workflow 节点类型为 "Agent Task" | 自动调用 AgentMesh 执行任务，结果写回 Workflow 上下文 |
| **Agent→目标** | Agent 完成分析任务 | 分析结果自动生成目标备注、偏差分析报告 |
| **因果链联动** | 上游目标进度变化 > 15% | 自动通知下游目标负责人并建议调整策略 |
