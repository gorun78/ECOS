2. 侧边栏"知识图谱"独立入口指向的是RAG检索页(AIPKnowledgeView)，而非真正的图谱浏览
3. ObjectKgSyncService已经在跑，但没有管理界面——哪些Object同步了？同步了多少？状态如何？
4. 三个不同的Neo4j连接(Neo4jQueryService/Neo4jGraphService/ObjectKgSyncService各自new Driver)，资源浪费

## 二、设计目标

将Neo4j知识图谱/数据图谱能力以**两个新Tab**的方式融入知识工作台，不改现有Tab结构：

- **Tab 7: 图谱同步** — 管理"哪些数据对象要同步到Neo4j图谱"，控制同步策略
- **Tab 8: 图谱探索** — 图形化浏览/搜索/分析Neo4j图谱

## 三、前端设计

### 3.1 KnowledgeView 新增两个Tab

```typescript
const [activeSubTab, setActiveSubTab] = useState<
  'architecture' | 'sync' | 'lineage' | 'ontology' | 'index' | 'rag' 
  | 'graph_sync' | 'graph_explore'  // ← 新增
>('architecture');
```

### 3.2 Tab 7: 图谱同步 (GraphSyncTab)

**功能**:
- 列出所有Object类型及其图谱同步状态（已同步数/未同步数/最近同步时间）
- 每类Object可独立开关同步
- "全量同步"按钮——手动触发全局Object→Neo4j同步
- 同步历史日志（最近10条成功/失败记录）

**API对接**:
- `GET /api/knowledge/sync/status` — 各Object类型同步状态
- `POST /api/knowledge/sync/trigger` — 触发全量同步
- `POST /api/knowledge/sync/object/{type}` — 触发单类Object同步

### 3.3 Tab 8: 图谱探索 (GraphExplorerTab)

**功能**:
- **全图视图**: 力导向图展示所有节点+关系，支持缩放/拖拽/点击
- **搜索框**: 输入关键词搜索节点，点击跳转到子图
- **节点详情面板**: 点击节点→右侧滑出属性面板
- **路径分析**: 选择两个节点→计算最短路径→高亮路径
- **邻居展开**: 点击节点→展开N度邻居（可配置1-3度）
- **领域过滤**: 下拉选择domain(本体域/数据域/业务域)筛选图谱
- **新建节点/边**: 轻量创建（节点名称+标签+属性，边=连接两个节点+类型）

**API对接**（已有，无需新建）:
- `GET /api/knowledge/graph?domain=xxx` — 全图数据
- `GET /api/knowledge/nodes/{id}` — 节点详情
- `GET /api/knowledge/search?q=xxx` — 搜索
- `GET /api/knowledge/path?s=xxx&t=yyy` — 路径
- `GET /api/knowledge/neighbors/{id}?d=2` — 邻居
- `POST /api/knowledge/nodes` — 创建节点
- `POST /api/knowledge/edges` — 创建边

**可视化库选择**: 复用现有 `GraphCanvas.tsx` 或 `OntologyGraph.tsx`，避免引入新依赖。

## 四、后端改进

### 4.1 Neo4j连接池统一

当前问题：三个Service各自new Driver，浪费连接。

**方案**: 提取 `Neo4jConfig` 为公共Bean，注入到各Service:

```java
@Configuration
public class Neo4jConfig {
    @Bean
    @ConditionalOnProperty("neo4j.uri")
    public Driver neo4jDriver(@Value("${neo4j.uri}") String uri, ...) {
        return GraphDatabase.driver(uri, AuthTokens.basic(user, pass), config);
    }
}
```

三个Service改为 `@Autowired Driver driver`，共享连接池。

### 4.2 新增同步管理端点

在 `KnowledgeGraphController` 或新建 `GraphSyncController`:

```java
// GET /api/knowledge/sync/status — 同步状态总览
// POST /api/knowledge/sync/trigger — 触发全量同步
// POST /api/knowledge/sync/object/{type} — 触发单类Object同步
// GET /api/knowledge/sync/logs — 同步历史
```

实现思路：`ObjectKgSyncService` 增加统计方法，记录每次同步的时间和数量。

### 4.3 模块归宿

| 文件 | 当前位置 | → 目标位置 | 理由 |
|------|---------|-----------|------|
| `Neo4jQueryService` | runtime-core | **dccheng** | 知识图谱专用，应隶属dccheng |
| `KnowledgeGraphService` | runtime-core | **dccheng** | 同上 |
| `KnowledgeGraphController` | dccheng | **dccheng** (不变) | 已在正确位置 |
| `ObjectKgSyncService` | workspace | **dccheng** | 同步逻辑应在知识引擎侧 |
| `Neo4jGraphService` | worldmodel | **dccheng** | IGraphService实现，统一管理 |
| `OntologyKgSyncService` | worldmodel | **dccheng** | 本体同步也归入知识域 |
| `PgGraphService` | worldmodel | **dccheng** | PG回退实现 |

> 注: 此调整与「后端架构重整-PMO指令」中阶段4(认知引擎)对齐——dccheng知识模块是cognitive-engine的来源之一。
