# ECOS 后端架构重整分析报告

> PMO 统筹分析 | 2026-06-24 | 输入：用户3个架构问题 + 代码审计数据

## 一、现状诊断

### 1.1 模块全景

```
databridge-v2/  (13个Maven模块)
├── databridge-common        — 公共基础 (ApiResponse, 工具类)
├── databridge-runtime       — 运行时核心 (ISystemDatabaseAccess, JdbcTemplate)
├── databridge-sysman        — 系统管理 (sysman-boot + sysman-impl) ← 唯一启动入口
├── databridge-aimod         — Agent引擎
├── databridge-buszhi        — 数据治理 (Pipeline/DQ)
├── databridge-datanet       — 数据网络 (Catalog/Metadata/DataSource)
├── databridge-dccheng       — 数据编制 (Ontology/KG/Glossary)
├── databridge-market        — 数据市场
├── databridge-portal        — 统一门户
├── databridge-workspace     — 工作空间
├── databridge-worldmodel    — 世界模型
├── databridge-gateway       — ⚠️ API网关 (空壳，0个Controller)
└── ecos-kanban              — 看板模块
```

### 1.2 关键事实

| 指标 | 数值 |
|------|:--:|
| Controller 总数 | **52** |
| Endpoint 总数 | **305** |
| 启动入口 | **仅1个** (sysman-boot) |
| API 路径前缀 | **17种** (`/api`, `/api/v1/ecos`, `/api/marketplace`, `/datanet`, ...) |
| 前端 Vite proxy | 硬编码 if-else 链，覆盖 6 种前缀 |
| api-gateway 模块 | **完全空壳** — 无 @SpringBootApplication，无 @RestController |

### 1.3 当前架构模式

```
前端 (Vite :5173)
    │
    │  Vite proxy (硬编码 rewrite)
    │  /api/v1/* → 直通
    │  /api/marketplace/* → 直通
    │  /api/portal/* → 直通
    │  其他 /api/* → /api/v1/gsxk/*
    ▼
sysman-boot :8081 (单体启动)
    │
    │  @ComponentScan 扫描全部模块
    ▼
52个Controller → PostgreSQL
```

**核心问题**：名义上是多模块架构，实际上是**单进程单体**。`api-gateway` 模块形同虚设，路由逻辑散落在 Vite proxy 的硬编码 rewrite 和 Spring Security 白名单中。

## 二、问题一：API Gateway 统一注册

### 2.1 现状问题

1. **路由散落三处**：Vite proxy (前端) + SecurityConfig 白名单 (后端) + 各 Controller 的 @RequestMapping — 新增一个模块需改 3 个地方
2. **无统一契约**：17 种前缀无文档、无 schema 校验、无版本管理
3. **CLI 开发受阻**：未来 CLI 前端需逐个适配前缀，无法通过统一 OpenAPI 文档自动生成客户端
4. **datanet 模块特殊**：使用 `/datanet/*` 前缀，与其他 `/api/*` 不一致

### 2.2 推荐方案：复用并激活 api-gateway

```
                     ┌─────────────────┐
CLI / Web / SDK  ←──│  api-gateway     │──→ Spring Cloud Gateway
                     │  :8080           │    统一路由表 + OpenAPI doc
                     └────────┬────────┘
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
         sysman-boot     portal-impl    datanet-controller
         :8081           (同进程)       (同进程)
```

**分三阶段落地**：

| 阶段 | 内容 | 工期 |
|------|------|:--:|
| **Phase A** (1周) | api-gateway 添加 `spring-cloud-gateway` 依赖，定义路由表 YAML，前端 Vite proxy 改为单一 `/api` → gateway 转发 | 3天 |
| **Phase B** (1周) | 生成 OpenAPI 3.0 文档（SpringDoc 自动扫描 52 个 Controller），前端 api.ts 基于 OpenAPI 生成类型 | 2天 |
| **Phase C** (后续) | CLI 前端通过 OpenAPI 文档自动生成客户端 SDK | 按需 |

**Phase A 最小改动**：
```yaml
# api-gateway/src/main/resources/application.yml
spring:
  cloud:
    gateway:
      routes:
        - id: sysman
          uri: http://localhost:8081/sys-man
          predicates: Path=/api/v1/**
        - id: portal
          uri: http://localhost:8081/sys-man
          predicates: Path=/api/portal/**
        # ... 其余路由
```

**利弊**：
- ✅ 前端只需一个 proxy target，不再维护 if-else
- ✅ 新增模块只需加一条路由配置
- ✅ 天然支持限流、日志、认证统一拦截
- ⚠️ 增加一跳网络延迟（本地 ~1ms，可忽略）
- ⚠️ api-gateway 本身成为单点 — 但当前就是单点，无回归

## 三、问题二：架构重组

### 3.1 外部组件统一接入层

当前状态：Doris/MinIO/Redis/Kafka/Neo4j/OPA/Elasticsearch — **全部仅在设计文档中，零行代码引用**。

**建议**：在 `databridge-common` 下增加 `connector` 子包，采用 **SPI 插件机制**：

```
databridge-common/
└── connector/
    ├── spi/
    │   ├── StorageConnector.java     (interface)
    │   ├── CacheConnector.java       (interface)
    │   ├── SearchConnector.java      (interface)
    │   └── GraphConnector.java       (interface)
    ├── postgres/   (默认实现)
    ├── redis/      (Phase 2)
    ├── minio/      (Phase 3)
    ├── neo4j/      (Phase 2)
    ├── opa/        (Phase 2)
    └── doris/      (Phase 3)
```

**原则**：
- 默认使用 PostgreSQL 实现全部接口（当前状态不变）
- Phase 2 引入 Neo4j 时，只需新增 `GraphConnector` 的 Neo4j 实现，业务代码零改动
- Phase 3 引入 MinIO 时，只需新增 `StorageConnector` 的 MinIO 实现

### 3.2 流程设计器统一

**当前设计器清单**：

| 设计器 | 页面 | 端点 | 生命周期 |
|--------|------|:--:|:--:|
| PipelineBuilder | DAG 编辑器 | `/api/v1/ecos/workflows` 14端点 | ⚠️ 缺 Create |
| WorkflowDesigner | 工作流设计器 | 同 PipelineBuilder | ⚠️ 缺 Create |
| OntologyDesigner | 本体设计器 | `/api/v1/ecos/ontologies` 16端点 | ✅ 较完整 |
| CodeWorkbook | 代码工作簿 | 前端 mock | ⚠️ 纯前端 |

**发现问题**：
1. PipelineBuilder 和 WorkflowDesigner **共享同一组 API**（`/api/v1/ecos/workflows`），功能边界模糊
2. CodeWorkbook 无后端，纯前端模拟
3. 四个设计器各用不同 UI 范式（PipelineBuilder 用 GraphCanvas，WorkflowDesigner 另起炉灶）

**建议统一方案**：

```
统一流程引擎 (Workflow Engine)
├── 核心 API: /api/v1/workflows
│   ├── POST   /definitions        (创建)
│   ├── PUT    /definitions/:id    (编辑)
│   ├── DELETE /definitions/:id    (删除)
│   ├── GET    /definitions/:id    (获取)
│   ├── POST   /definitions/:id/publish  (发布)
│   ├── POST   /definitions/:id/execute  (执行)
│   └── GET    /instances          (运行实例)
│
├── PipelineBuilder   ── 数据管道子类型 (type=pipeline)
├── WorkflowDesigner  ── 审批流子类型  (type=approval)
├── OntologyDesigner  ── 本体子类型    (type=ontology)
└── CodeWorkbook      ── 代码子类型    (type=code)
```

**统一生命周期**（所有设计器共用）：
```
DRAFT → PUBLISHED → RUNNING → COMPLETED/FAILED
  ↓         ↓
ARCHIVED  DEPRECATED
```

### 3.3 插件机制（企业规模适配）

**需求本质**：同一套代码，中小企业只需基础功能，大型企业需要全量功能。

**推荐：Spring Boot 条件装配 + 功能开关**

```java
// 基于系统配置表动态启用/禁用模块
@Configuration
@ConditionalOnConfig(key = "plugin.pipeline.enabled", havingValue = "true")
public class PipelineAutoConfiguration {
    // 仅在配置启用时加载 Pipeline 相关 Bean
}
```

**实现路径**：

| 规模 | 启用模块 | 配置 |
|------|----------|------|
| **小型**（<100人） | runtime + sysman + portal | `plugin.*.enabled=false` |
| **中型**（100-1000人） | + datanet + dccheng + buszhi | `plugin.pipeline.enabled=true` |
| **大型**（1000+人） | + aimod + worldmodel + market | 全部启用 |

**前端对应**：
- 后端 `/api/v1/system/plugins` 返回已启用的插件列表
- 前端 Sidebar 根据插件列表动态渲染菜单
- 禁用的模块不注册路由

**与系统配置模块联动**（已有 infra）：
```
sys_config 表新增:
  plugin.pipeline.enabled    = true
  plugin.agent_mesh.enabled  = false
  plugin.world_model.enabled = true
  ...
```

## 五、决策建议

| 问题 | 建议 | 理由 |
|------|------|------|
| api-gateway 是否立即激活 | ✅ 是 | 改动小（加依赖+路由YAML），收益大（前端简化+CLI就绪+统一文档） |
| Connector SPI 是否立即做 | ⚠️ 仅接口 | 只定义接口不写实现，Phase 2 引入 Neo4j 时不需改业务代码 |
| 流程设计器合并 | ✅ 是 | 当前 4 个设计器各自为政，统一后可复用 GraphCanvas + 生命周期 |
| 插件机制 | 先建开关 | 利用已有 sys_config 表，成本极低，Phase 2 全面启用 |
