# ECOS Demo: 供应商准入智能审查

> **演示日期**：2026-06-13  
> **模型**：DeepSeek v4-flash  
> **后端**：Databridge Sysman (Java 17 + Spring Boot + MyBatis)  
> **前端**：C2EOS (React 19 + Vite 6)  
> **访问地址**：http://localhost:3000/agent-studio

---

## 场景概述

**业务背景**：采购部门收到一家新供应商「星辰科技有限公司」的入库申请，需要完成准入审查。

**演示目标**：在 5 分钟内让一个不了解 ECOS 的人看懂「Agent 驱动的企业操作系统」是什么意思——Agent 不是简单地查数据库或查文档，而是结合两者做推理判断，并最终触发业务流程。

**演示流程**：5 步，逐步展示 Agent 的 5 种核心能力（object_query → ontology_explore → knowledge_search → 复合推理 → workflow_start），全程在同一个 Agent Studio 会话中完成。

## 技术亮点

### 1. 五种能力的协同编排

```
object_query ──→ 查结构化数据（供应商档案）
ontology_explore ──→ 探索数据模型（Supplier 关联关系）
knowledge_search ──→ 检索文档知识（准入管理办法）
复合推理 ──→ 数据+知识融合判断（逐项对比→结论）
workflow_start ──→ 触发业务流程（创建审核实例）
```

### 2. 复合推理的数据+知识融合

第 4 步是整个 Demo 的技术高光：
- Agent **自主决策**需要同时查询结构化数据（object_query）和非结构化知识（knowledge_search）
- 在 5 次工具调用中完成「查数据→读政策→逐项对比→综合判断」
- 最终给出有具体理由的明确结论，而非「缺乏信息无法判断」

### 3. ReAct 引擎的实时 Trace

每一步的推理过程通过 SSE 实时推送到前端：
- 🤔 Thought：Agent 的推理思路
- 🔧 Tool Call：调用的工具和参数
- 📊 Observation：工具返回的结果
- ✅ Final：最终回答

### 4. Agent Node 的 decision_routing

第 5 步启动的工作流包含 AGENT_NODE 配置：
- `system_prompt`：定义了供应商审核专家的角色
- `tools`：可用工具集（object_query, ontology_explore, knowledge_search）
- `output_mapping`：将 Agent 输出映射到工作流变量
- `decision_routing`：根据 risk_level 自动路由到不同终态

### 5. 三层记忆的跨步骤上下文保持

同一 session 内的 5 步共享记忆上下文：
- **工作记忆（Working Memory）**：当前 session 的活跃实体和工作流
- **短期记忆（Short-term）**：最近 3 条问答历史
- **长期记忆（Long-term）**：跨 session 的持久化事实（按重要性评分）
