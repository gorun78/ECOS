## 一、架构概览

### 1.1 分层架构

```
┌─────────────────────────────────────────────────────────────┐
│  前端 (Monaco / React Flow / ECharts)                       │
├─────────────────────────────────────────────────────────────┤
│  新 Controller 层 (9 个)                                     │
│  ├── QueryController          /api/v1/engine/data/query/*   │
│  ├── SyncTaskController       /api/v1/engine/data/sync/*    │
│  ├── PipelineTaskController   /api/v1/engine/data/pipeline/* │
│  ├── QualityController        /api/v1/engine/data/quality/* │
│  └── CleanseController        /api/v1/engine/data/cleanse/* │
│                                                              │
│  + 现有 5 个 Controller 保持不变                              │
├─────────────────────────────────────────────────────────────┤
│  新 Service 层 (8 个) — engine/data-engine/data-engine-impl/ │
│  ├── QueryExecutionService      ← ConnectorFactory + Jdbc   │
│  ├── SyncTaskService            ← TaskScheduler + Connector │
│  ├── PipelineIntegrationService ← PipelineService + trans.. │
│  ├── QualityInspectionService   ← ConnectorFactory + Alert  │
│  ├── CleanseExecutionService    ← ConnectorFactory + Alert  │
│  ├── SyncRunService             ← 执行记录 & 日志管理        │
│  ├── PipelineRunService         ← DAG 执行 & 断点续跑       │
│  └── QualityDashboardService    ← 质量总览聚合              │
│                                                              │
│  + 现有 4 个 Service 保持不变                                 │
├─────────────────────────────────────────────────────────────┤
│  基础设施层 (复用现有)                                        │
│  ├── ConnectorFactory → JdbcConnector (PostgreSQL/MySQL/…)  │
│  ├── JdbcTemplate (构造器注入，仅在 Service 层)               │
│  ├── runtime TaskSchedulerService (cron 调度)                │
│  ├── datanet PipelineService (DAG 定义 & transform 算子)     │
│  └── runtime IAlertService (质量告警 & 清洗通知)             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 四能力与模块映射

| # | 能力 | Controller | Service | 复用基础设施 |
|---|------|-----------|---------|------------|
| 1 | 统一数据访问层 | `QueryController` | `QueryExecutionService` | ConnectorFactory + JdbcTemplate |
| 2 | 数据同步任务 | `SyncTaskController` | `SyncTaskService` + `SyncRunService` | TaskScheduler + ConnectorFactory |
| 3 | 数据管道任务 | `PipelineTaskController` | `PipelineIntegrationService` + `PipelineRunService` | PipelineService + transform 算子 |
| 4 | 数据质量/清洗 | `QualityController` + `CleanseController` | `QualityInspectionService` + `CleanseExecutionService` + `QualityDashboardService` | ConnectorFactory + IAlertService |

## 二、数据库 Schema DDL（14 张新表）

### 2.1 设计原则

- 所有新表统一 `ecos_` 前缀，与现有 `td_` 前缀表共存
- 主键使用 UUID (varchar(36))
- `created_at` / `updated_at` 使用 `TIMESTAMP DEFAULT CURRENT_TIMESTAMP`
- 任务定义类表使用 `config_json JSONB` 存储 DSL 内容
- 执行记录类表使用 `status` + `error_msg` + `retry_count` 模式

### 2.2 统一查询模块（2 张表）

```sql
-- ============================================================
-- 1. ecos_query_template — 查询模板（参数化 SQL 片段）
-- ============================================================
CREATE TABLE ecos_query_template (
    id              VARCHAR(36)  PRIMARY KEY,
    name            VARCHAR(200) NOT NULL,                  -- 模板名称
    description     TEXT,                                   -- 描述
    datasource_id   VARCHAR(36)  NOT NULL,                  -- 关联数据源 ID
    sql_content     TEXT         NOT NULL,                  -- SQL 模板内容（含 :param 占位符）
    params_json     JSONB        DEFAULT '{}'::jsonb,       -- 参数定义 [{"name":"start_date","type":"date","default":"2026-01-01"}]
    timeout_seconds INTEGER      DEFAULT 30,                -- 超时秒数
    max_rows        INTEGER      DEFAULT 10000,             -- 最大返回行数
    created_by      VARCHAR(100),                           -- 创建人
    created_at      TIMESTAMP    DEFAULT CURRENT_TIMESTAMP,
    updated_at      TIMESTAMP    DEFAULT CURRENT_TIMESTAMP
);

-- ============================================================
-- 2. ecos_query_history — 查询历史
-- ============================================================
CREATE TABLE ecos_query_history (
    id              VARCHAR(36)  PRIMARY KEY,
    datasource_id   VARCHAR(36)  NOT NULL,                  -- 数据源 ID
    datasource_name VARCHAR(200),                           -- 数据源名称（冗余，便于展示）
    sql_content     TEXT         NOT NULL,                  -- 执行的 SQL
    execution_time_ms INTEGER,                              -- 执行耗时（毫秒）
    row_count       INTEGER,                                -- 返回行数
    status          VARCHAR(20)  NOT NULL DEFAULT 'SUCCESS', -- SUCCESS / ERROR / TIMED_OUT / CANCELLED
    error_msg       TEXT,                                   -- 错误信息
    template_id     VARCHAR(36),                            -- 关联模板 ID（可选）
    executed_by     VARCHAR(100),                           -- 执行人
    executed_at     TIMESTAMP    DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_qh_ds     ON ecos_query_history(datasource_id);
CREATE INDEX idx_qh_user   ON ecos_query_history(executed_by);
CREATE INDEX idx_qh_time   ON ecos_query_history(executed_at DESC);
```

## 三、Service 接口签名

> 所有 Service 实现类放在 `engine/data-engine/data-engine-impl/src/main/java/com/chinacreator/gzcm/engine/data/service/`
> 所有 Service 接口定义在 `engine/data-engine/data-engine-api/src/main/java/com/chinacreator/gzcm/engine/data/`

### 3.1 QueryExecutionService（统一查询）

```java
package com.chinacreator.gzcm.engine.data;

import java.util.List;
import java.util.Map;

/**
 * 统一 SQL 查询执行服务。
 * 通过 ConnectorFactory 获取对应数据源的 Connector，在目标库执行 SQL。
 */
public interface QueryExecutionService {

    /**
     * 执行 SQL 查询。
     * @param datasourceId 数据源 ID
     * @param sql          SQL 语句
     * @param params       参数映射（:param 占位符替换）
     * @param timeoutSeconds 超时秒数
     * @param maxRows      最大返回行数
     * @return {columns, rows, rowCount, elapsedMs, datasourceName}
     */
    Map<String, Object> execute(String datasourceId, String sql,
                                Map<String, Object> params,
                                int timeoutSeconds, int maxRows);

    /**
     * 获取 Schema 树结构（数据源 → Schema → 表 → 字段）。
     * 使用 Connector.getSchemaInfo() + JdbcTemplate 的 DatabaseMetaData。
     */
    List<Map<String, Object>> getSchemaTree(String datasourceId);

    /**
     * 保存查询模板。
     */
    Map<String, Object> saveTemplate(Map<String, Object> template);

    /**
     * 查询模板列表。
     */
    List<Map<String, Object>> listTemplates(String keyword);

    /**
     * 获取单个模板。
     */
    Map<String, Object> getTemplate(String id);

    /**
     * 删除模板。
     */
    void deleteTemplate(String id);

    /**
     * 查询执行历史（分页）。
     */
    Map<String, Object> getQueryHistory(String datasourceId, String status,
                                         int page, int pageSize);

    /**
     * 取消正在执行的查询。
     */
    void cancelQuery(String historyId);
}
```

```java
/**
 * 同步任务生命周期管理。
```

## 五、任务状态机

### 5.1 统一任务状态机

适用对象：`ecos_sync_run`、`ecos_pipeline_run`、`ecos_pipeline_step_run`、`ecos_cleanse_run`

```
                    ┌─────────┐
                    │ PENDING │  ← 初始状态（任务创建，等待调度）
                    └────┬────┘
                         │ schedule / trigger
                         ▼
                    ┌─────────┐
                    │ QUEUED  │  ← 已提交到 TaskScheduler 队列
                    └────┬────┘
                         │ dequeue & start
                         ▼
              ┌─────────────────────┐
              │      RUNNING        │  ← 正在执行
              └──┬──────┬──────┬───┘
                 │      │      │
        ┌────────┘      │      └──────────┐
        ▼               ▼                 ▼
   ┌──────────┐   ┌──────────┐     ┌───────────┐
   │SUCCEEDED │   │  FAILED  │     │ CANCELLED │ ← 手动取消
   └──────────┘   └────┬─────┘     └───────────┘
                       │
                  ┌────┴────┐
                  │ 检查重试  │
                  │ 次数<N?  │
                  └────┬────┘
                       │ YES
                       ▼
                  ┌──────────┐
                  │ RETRYING │ ← 等待退避时间后重置为 QUEUED
                  └──────────┘
                       │ NO → 最终状态 FAILED
                       ▼
                  ┌──────────┐
                  │TIMED_OUT │ ← 超时未完成（最终状态）
                  └──────────┘
```

### 5.2 状态转换表

| 源状态 | 目标状态 | 触发条件 | 说明 |
|--------|---------|---------|------|
| `PENDING` | `QUEUED` | `TaskScheduler.scheduleTask()` | 提交到调度队列 |
| `QUEUED` | `RUNNING` | 调度器分配执行 | 开始消费数据 |
| `RUNNING` | `SUCCEEDED` | 同步成功完成 | 记录 rows + elapsedMs |
| `RUNNING` | `FAILED` | 执行异常 | 记录 errorMsg |
| `RUNNING` | `CANCELLED` | 用户手动取消 | 发送 interrupt |
| `RUNNING` | `TIMED_OUT` | 超过 timeout 秒 | 记录超时信息 |
| `FAILED` | `RETRYING` | retryCount < maxRetry | 等待 backoffSeconds |
| `RETRYING` | `QUEUED` | 退避结束，重新入队 | 增加 retryCount |
| `QUEUED` | `CANCELLED` | 用户取消排队中的任务 | 从调度队列移除 |

### 5.3 管道步骤子状态机

适用对象：`ecos_pipeline_step_run`

```
PENDING → RUNNING → SUCCEEDED
                  → FAILED → (管道整体状态变为 FAILED)
                  → SKIPPED (依赖步骤失败，跳过执行)
```

### 5.4 同步任务生命周期

```
创建 → 已保存(PENDING cron) ──┬── 手动触发 → QUEUED → RUNNING → ...
                              │
                              └── cron 触发 → QUEUED → RUNNING → ...

暂停 → cron 调度取消，但任务定义保留，手动触发仍然可用
恢复 → 重新注册 cron 到 TaskScheduler
删除 → 取消 cron + 软删除/硬删除记录
```

## 六、与现有模块的集成方案

### 6.1 复用 ConnectorFactory + JdbcTemplate

**原则**：所有对异构数据源的访问统一走 `ConnectorFactory`，内部 SQL 操作走 `JdbcTemplate`。

```
┌────────────────────────────────────────────────────┐
│  Service 层                                         │
│                                                     │
│  QueryExecutionService                              │
│    ├── 语法校验 / SQL 参数化 → JdbcTemplate         │
│    └── 执行查询 → ConnectorFactory.getConnector()   │
│                  → connector.query(sql, params)      │
│                                                     │
│  SyncTaskService / SyncExecutionEngine              │
│    ├── 源端查询 → ConnectorFactory (source ds)      │
│    ├── 字段映射/表达式转换 → transform 算子          │
│    └── 目标写入 → ConnectorFactory (target ds)      │
│                                                     │
│  QualityInspectionService                           │
│    ├── 质量规则执行 → ConnectorFactory → JDBC       │
│    └── 统计查询（COUNT/GROUP BY/条件筛选）           │
│                                                     │
│  CleanseExecutionService                            │
│    ├── 执行清洗 → ConnectorFactory → JDBC           │
│    └── UPDATE/SET 操作 + 备份表创建                  │
└────────────────────────────────────────────────────┘
```

**ConnectorFactory 扩展点**：如需支持新数据源类型（如 ClickHouse），只需新增一个实现 `Connector` 接口的 Bean，工厂自动发现。

### 6.2 同步任务集成 — runtime TaskScheduler

```
SyncTaskService                     TaskSchedulerService
     │                                      │
     ├─ createTask()                        │
     │   └── 保存到 ecos_sync_task           │
     │   └── 若 cron_expression 非空         │
     │       └── scheduleTask(              │
     │           TaskDescription{           │
     │             taskId: sync-task-{id},  │
     │             taskType: "SYNC",        │
     │             taskConfig: JSON DSL,    │
     │             retryCount: 3,          │
     │             timeout: 600000         │
     │           },                         │
     │           cron_expression            │
     │         )                            │
     │                                      │
     ├─ triggerRun()                        │
     │   └── scheduleTask(TaskDescription)  │ ← 立即执行
     │                                      │
     ├─ pauseTask()                         │
     │   └── cancelSchedule(scheduleId)     │ ← 取消 cron
     │                                      │
     └─ resumeTask()                        │
         └── scheduleTask(..., cron)        │ ← 重新注册
```

**SyncExecutionEngine（内部类）** 实现 `Runnable`，在 `TaskScheduler` 触发时执行实际的同步逻辑：
1. 从 `ecos_sync_task.config_json` 读 DSL
2. 创建 `ecos_sync_run` 记录（status=PENDING → QUEUED → RUNNING）
3. 通过 `ConnectorFactory` 连接源数据源，执行 source query
4. 进行字段映射 + 冲突解决策略
5. 通过 `ConnectorFactory` 写入目标数据源
6. 更新 `ecos_sync_run` 状态 + 日志
7. 执行 preSync / postSync 钩子

### 6.3 管道任务集成 — datanet PipelineService

```
PipelineIntegrationService              PipelineService (datanet-api)
     │                                        │
     ├─ createTask()                          │
     │   ├── 保存到 ecos_pipeline_task         │
     │   └── 调用 createDefinition({          │
     │         name, nodes: steps[],          │
     │         edges: input/output 连线       │
     │       })                                │
     │                                        │
     ├─ triggerRun()                          │
     │   └── PipelineExecutionEngine          │
     │       ├── 解析 ecos_pipeline_step 列表  │
     │       ├── 按 DAG 拓扑排序               │
     │       └── 逐步骤调用 transform 算子     │
     │           ├── extract → ConnectorFactory│
     │           ├── filter → transform 算子   │
     │           ├── transform → transform 算子│
     │           ├── join → transform 算子     │
     │           ├── aggregate → transform 算子│
     │           ├── sort → transform 算子     │
     │           ├── limit → transform 算子    │
     │           └── load → ConnectorFactory   │
     └── 中间结果存储在临时表/jobContext        │
```

**8 种 transform 算子** 直接复用 datanet 已注册的 `Transform` 实现（通过 `PipelineService` 已有的节点执行器），无需新增算子注册机制。

**断点续跑实现**：
- 每个步骤执行前检查 `ecos_pipeline_step_run` 中是否有已成功的记录
- 若 `stepRun.status == 'SUCCEEDED'`，跳过该步骤
- 若 `stepRun.status == 'FAILED'`，从该步骤重新开始

### 6.4 质量/清洗集成 — runtime IAlertService

```
QualityInspectionService              IAlertService (runtime-core)
     │                                       │
     ├─ runInspection()                      │

     │   ├── 计算评分: (passRows/totalRows)*100│
     │   ├── 写入 ecos_quality_report          │
     │   └── 若 score < alertMinScore         │

     │           )                            │

CleanseExecutionService               IAlertService
     │                                       │

     │   ├── 可选：创建备份表                  │
     │   ├── 逐行执行清洗规则                  │

     │             ruleId,                    │

     │           )                            │
```

**告警规则预创建**：Spring Boot 启动时，通过 `@PostConstruct` 在 IAlertService 中预注册两类告警规则：

2. `CLEANSE_LARGE_CHANGE` — 清洗大变更告警

## 八、实施建议

### 8.1 分阶段实施（对齐 PRD Sprint）

| Sprint | 交付内容 | 涉及表 | 涉及 Service |
|--------|---------|--------|-------------|
| **Sprint 2.1** | 统一查询 + Schema 浏览 | `ecos_query_template`, `ecos_query_history` | `QueryExecutionService` |
| **Sprint 2.2** | 同步任务 | `ecos_sync_task`, `ecos_sync_run` | `SyncTaskService`, `SyncRunService` |
| **Sprint 2.3** | 管道任务 | `ecos_pipeline_task`, `ecos_pipeline_step`, `ecos_pipeline_run`, `ecos_pipeline_step_run` | `PipelineIntegrationService`, `PipelineRunService` |
| **Sprint 2.4** | 数据质量 | `ecos_quality_rule`, `ecos_quality_report`, `ecos_quality_issue` | `QualityInspectionService`, `QualityDashboardService` |
| **Sprint 2.5** | 数据清洗 | `ecos_cleanse_rule`, `ecos_cleanse_run`, `ecos_cleanse_change_log` | `CleanseExecutionService` |
| **Sprint 2.6** | 全链路集成 + E2E | — | 各 Service 间协作 |

### 8.2 开发规范

1. **DDL 执行**：通过项目内 SQL 脚本手动执行，Flyway 已禁用
2. **Controller 规范**：构造器注入 Service，返回 `ApiResponse.success()` / `.error()`
3. **异常处理**：统一抛出 `DataBridgeException` 子类，由 `GlobalExceptionHandler` 捕获
4. **Schema 变更原则**：V2 新表统一 `ecos_` 前缀，只加不删列/表
5. **测试**：每个 Service 对应一个 `*Test.java`，使用 `@SpringBootTest`

### 8.3 关键风险

| 风险 | 缓解措施 |
|------|---------|
| Connector 不支持某数据源的 schema 查询 | 先用 JdbcTemplate + DatabaseMetaData 兜底，再补充 Connector 实现 |
| TaskScheduler 的 cron 实现与数据库 cron 不一致 | 验证 TaskSchedulerServiceImpl 的 cron 解析器，确保兼容标准 cron 表达式 |
| PipelineService transform 算子不支持某 DSL 操作 | 在 PipelineExecutionEngine 中针对 8 种类型分别实现，不依赖 PipelineService 的通用执行器 |
| 大批量同步时内存溢出 | 采用流式读取（ResultSet fetchSize）+ 分批写入（batchSize 1000） |
