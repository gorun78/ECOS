## 阶段三：Pipeline 2.0 后端实现

### 指令 3.1 — 并行实现 BE (5 Controller + 6 Service) + FE (6 组件)

两路 Agent 并行，10 分钟内产出 11 Java + 9 TSX。

**BE 产出 (data-engine-impl)**:
```
Controller:  PipelineFunctionController  (函数注册表 CRUD)
             PipelineTaskController      (14端点: CRUD+run+cancel+runs)
```

**FE 产出 (pipeline-editor/)**:

### 指令 3.2 — 修复编译错误

| 错误 | 修复 |
|------|------|
| `int != null` — 基本类型与 null 比较 | `count != null && count >= 120` → `count >= 120` |
| `Thread.ofVirtual()` — JDK21 API | → `new Thread(...).start()` |

### 🔧 改进建议 3

> Controller 直接使用 JdbcTemplate 违反 Iron Law 5 ("Controller 禁止直接用 JdbcTemplate，必须走 Service 层")。Pipeline 的函数注册/任务管理都是 Controller 直接调 JdbcTemplate。
> **建议**: 后续重构时提取 PipelineFunctionService/PipelineTaskService 等，Controller 只注 Service。

## 阶段四：Git 统一

### 指令 4.1 — 用户纠正: “Git 应使用 runtime 统一能力”

原有两套 GitService:
- `runtime-core/git/GitService` (5方法 + 内存桩)
- `sysman/config/git/GitService` (6方法)
- `data-engine/JGitServiceImpl` (独立实现)

### 指令 4.2 — 统一到 runtime-core

**执行**:
1. 删除 `data-engine/JGitServiceImpl.java`
2. 扩展 `runtime-core/git/GitService` 接口 (5 → 11方法)
3. 重写 `runtime-core/git/GitServiceImpl.java` (JGit 真正实现, 替换内存桩)
4. 更新 `PipelineGitController` 注入 runtime `GitService`
5. Pipeline 专用方法 (writePipelineYaml/loadPipelineYaml) 下沉为 Controller 私有方法
6. 添加 JGit 依赖到 runtime-core/pom.xml

### 🔧 改进建议 4

> 两套 GitService 接口差异大 — runtime 版用 `GitRepository` 实体，sysman 版用 String 参数。统一后 11 个方法混在一起。
> **建议**: 将 `GitService` 拆为两个接口: `GitCoreService` (原5方法用 GitRepository) + `GitPipelineService` (扩展6方法用 String)，但这又回到分裂。或者反过来，sysman 的 GitService 也改为调用 runtime 的统一实现。

### 指令 8.2 — 与 ontology-engine 的 LineageController 重名

**问题**: ontology-engine 已有 `LineageController`，同名 Spring Bean 冲突导致 Gateway 启动失败。

**修复**: 改名为 `DataLineageController`

### 验证结果

```
Pipeline A: 8 nodes, 8 edges ✅
Pipeline B: 7 nodes, 8 edges ✅
Build Topology: 15 nodes, 16 edges ✅
```

### 🔧 改进建议 8

> 多个引擎模块注册相同的 Controller 名称是隐患。当前 ontology-engine 和 data-engine 都有 LineageController。
> **建议**: Controller 命名规范加入模块前缀，如 `DataLineageController` / `OntologyLineageController`，避免 Spring Bean 冲突。或在不同引擎使用不同 RequestMapping 前缀。

## 通用改进建议汇总

### A. 开发流程

| 问题 | 建议 |
|------|------|
| 编译-重启-验证循环长 (每次 3-5 分钟) | 开发时用 `spring-boot:run` 的 JRebel/HotSwap，或降低 gateway 启动时间 |
| Controller 直接 JdbcTemplate | 强制 Service 层，Controller 只注 Service |
| 配置多源 (3 处需同步) | 统一到 YAML 文件驱动 |
| 新增 API 后忘记重启 | CI 流程中加入自动重启 + smoke test |

### B. 模块边界

| 问题 | 建议 |
|------|------|
| GitService 两套接口 | 最终统一到 runtime-core，sysman 的也迁过来 |
| LineageController 两套 | data-engine 的 Lineage API 应与 ontology-engine 的不同路径前缀区分 |
| Runtime-core 388 文件过重 | 按之前瘦身计划逐步迁移到各引擎 |

### C. 配置管理

| 问题 | 建议 |
|------|------|
| 默认值写死在 Java 静态 Map | 外置到 `defaults.yml` |
| DB 种子数据用硬编码数组 | 改为读 YAML → INSERT |
| 新增配置项易遗漏 | 配置项注册机制：定义一个 `@ConfigDefault` 注解，自动注册 |

### D. 测试

| 问题 | 建议 |
|------|------|
| 仅 curl 单端点测试 | 增加链式场景测试 (curl 脚本文件) |
| 无前端 E2E | Selenium/Playwright 脚本 |
| 环境依赖多 | Docker Compose 一键启动开发环境 |

### E. 知识管理

| 问题 | 建议 |
|------|------|
| PB 函数爬取失败 | 准备离线 JSON 清单 |
| 每次启动需手动 `POST /start` | Engine 自动启动 (已有 `@PostConstruct`，但未生效) |
