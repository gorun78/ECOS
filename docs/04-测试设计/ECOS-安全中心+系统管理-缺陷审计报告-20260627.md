## 问题 1：前端功能与操作手册存在差异

### 诊断

| 维度 | 操作手册描述 | 前端实际 |
|------|-------------|---------|
| 安全配置 | `/api/v1/security/profile` GET/PUT 含 `passwordMinLength`/`mfaEnabled`/`sessionTimeout` 等字段 | `SecurityConfigPanel.tsx` 使用的是 `clearanceLevel`/`linkedWorkstation`/`auditMode`/`sandboxMandatory` 四字段 |
| 用户管理 | 手册列出 CRUD + 密码重置 + 状态切换 | `UserManagement.tsx` 有 CRUD 但缺少密码重置 UI、缺少状态切换 UI、缺少用户-角色关联 UI |
| 组织管理 | 手册提及组织树 | `UserManagement.tsx` orgs tab 使用 `/organizations/all`（平铺列表），没用 `/organizations/tree` |
| 审计日志 | 手册详细 curl 命令 | `SecurityAudit.tsx` 存在但实际 API 调用需确认 |
| 字典管理 | 手册无 | `DictManager.tsx` 存在 |
| 系统配置 | 手册无 | `SystemConfigManager.tsx` 存在 |
| 租户管理 | 手册无 | `TenantManager.tsx` 存在（883行完整UI） |

**根因**: 操作手册基于 API 端点编写，但前端页面是独立开发的，两者未对齐。

### 修复方案
1. **FE-01**: 在 `SecurityConfigPanel.tsx` 中增加"密码策略"和"会话管理"两个子面板
2. **FE-02**: `UserManagement.tsx` 增加密码重置按钮、状态切换开关、用户-角色分配弹窗
3. **FE-03**: orgs tab 改用 `/organizations/tree` API + TreeView 组件
4. **PM-01**: 重写操作手册，以实际前端页面截图/功能为准

## 问题 2：组织机构应为树状，目前为平铺；用户应与组织关联

**后端已支持树状结构**:

**种子数据问题**:

### 修复方案
1. **FE-04**: `fetchOrgs()` 改用 `/api/v1/system/organizations/tree`，使用递归 TreeView 组件
2. **FE-05**: 用户创建/编辑表单增加组织选择器（TreeSelect）
3. **FE-06**: 用户列表增加"所属组织"列
4. **BE-01**: 增加种子组织数据（至少 3 级：集团→分公司→部门）

## 问题 3：缺少授权和鉴权过程

### 诊断

**当前状态**:
- JWT 认证 ✅（`JwtAuthenticationFilter` 正常工作）
- RBAC 数据模型 ✅（用户→角色→权限，三张关联表完整）
- 种子数据 ✅（20 条权限，admin/operator/user 三角色）
- **但是**: 
  - `JwtAuthenticationFilter` 仅验证 Token 有效性，**未检查用户权限**
  - 没有 `@PreAuthorize` / `@RequirePermission` 注解拦截
  - 前端路由无权限守卫（所有页面随便访问）
  - 安全配置（准入等级等）仅存在内存中，**从未应用于请求拦截**

### 修复方案
1. **BE-02**: 在 `JwtAuthenticationFilter` 中加载用户权限到 SecurityContext
2. **BE-03**: 创建 `@RequirePermission` 注解 + AOP 切面
3. **BE-04**: 核心 Controller 方法添加权限注解
4. **FE-07**: 前端路由守卫，根据用户角色过滤菜单

## 问题 4：用户、角色、权限种子数据存在乱码

### 诊断

**已确认**: 数据库中的中文数据全部乱码：

| 字段 | 实际存储 | 期望值 |
|------|---------|--------|
| ROLE_NAME (R001) | `绯荤粺绠＄悊鍛` | `系统管理员` |
| ROLE_NAME (R002) | `杩愮淮鎿嶄綔鍛` | `运维操作员` |
| DESCRIPTION (R001) | `绯荤粺绠＄悊鍛橈紝鎷ユ湁鎵?湁鏉冮檺` | `系统管理员，拥有所有权限` |

**根因**: `seed.sql` 文件本身是 UTF-8 编码，但在某个环节（可能是 Flyway 迁移时连接参数缺 `characterEncoding=UTF-8`）被误解释为 Latin-1 → 再编码为 UTF-8，产生"双重编码"乱码。

**影响**: 前端角色名显示乱码，用户体验差。

### 修复方案
1. **BE-05**: 重写 `seed.sql` INSERT 语句为纯 ASCII（或用 Flyway Java 迁移插入 UTF-8）
2. **BE-06**: 检查 Flyway 配置是否设置了 `flyway.encoding=UTF-8`
3. **DB-01**: 执行 UPDATE 直接修复已有乱码数据

## 问题 6：系统配置项在系统中的应用实现情况不明

### 诊断

**配置表**: `sys_config` (V13 Flyway)，Schema:
```
config_group, config_key, config_value, config_type, config_label
```

**API 层面**:
- `SysConfigController`: GET/PUT 可读写 ✅
- `ConfigController` (`/api/system/config/items`): 老接口，`@Autowired(required=false)` — **configService 可能为 null**

**消费者审计**:
- ❌ 无代码从 `sys_config` 读取密码策略并应用于实际密码验证
- ❌ 无代码从 `sys_config` 读取会话超时并应用于 JWT 过期
- ❌ 无代码从 `sys_config` 读取 MFA 配置
- ❌ 配置项的"实际生效路径"不透明

### 修复方案
1. **BE-08**: 创建 `SysConfigService` 统一读取入口 + 缓存
2. **BE-09**: 将密码策略/会话超时/MFA 等配置接入实际业务逻辑
3. **FE-09**: `SystemConfigManager.tsx` 增加"配置生效状态"指示（绿色/灰色）

## 问题 8：安全配置如何与角色/用户关联？准入等级分级？

### 诊断

**根本问题**: `SecurityConfigController` 是**全局单例**（内存 `ConcurrentHashMap`，只有一个 `isDefault=true` 的 profile），不与任何用户/角色关联。

| 安全字段 | 当前状态 | 期望 |
|---------|:--:|------|
| `clearanceLevel` (准入等级) | 全局一个值 | 按角色/用户分等级 |
| `linkedWorkstation` (物理工作站) | 全局一个值 | 按用户/角色绑定 |
| `auditMode` (审计力度) | 全局一个值 | 按用户/角色 + 资源类型 |
| `sandboxMandatory` (沙盒) | 全局 bool | 按角色/资源 |

### 准入等级建议设计

| 等级 | 名称 | 权限范围 |
|:--:|------|------|
| L0 | 公开 | 只读公开数据，无写入权限 |
| L1 | 内部 | 读内部数据 + 提交任务 |
| L2 | 保密 | 读保密数据 + 管理任务 + 查看审计 |
| L3 | 机密 | 全量读 + 管理用户/角色 + 修改配置 |
| L4 | 绝密 | 系统管理 + 安全配置 + 物理工作站绑定 |

### 修复方案
1. **BE-11**: 重构 `SecurityProfile` — 从全局单例改为 User/Role 级联配置
2. **BE-12**: 新增 `td_user_security_profile` 和 `td_role_security_profile` 表
3. **BE-13**: `SecurityConfigController` 支持按 userId/roleId 读写
4. **BE-14**: 实现准入等级拦截器 — 检查用户 clearanceLevel ≥ 资源所需等级
5. **FE-11**: `SecurityConfigPanel.tsx` 增加角色/用户选择器和准入等级滑块
6. **FE-12**: 角色列表增加"安全配置"入口（点击角色→配置其安全等级）
