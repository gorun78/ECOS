## 二、逐维度详细评审

### 2.1 技术选型成熟度与兼容性

#### 2.1.1 Neo4j 5 社区版 (Docker)

| 项目 | 评估 |
|------|------|
| **成熟度** | ⭐⭐⭐⭐⭐ Neo4j 5.x 自 2021 年 GA，截至 2026 年已极为成熟，社区版功能完整（Cypher 查询、APOC 扩展、多数据库）。全球数千企业生产使用。 |
| **Spring 兼容性** | ⭐⭐⭐⭐ Spring Data Neo4j 6.x/7.x 原生支持 Spring Boot 3.x，`@Node`/`@Repository` 注解体系完善。需新增依赖 `spring-boot-starter-data-neo4j`。 |
| **与现有 PG 知识图谱的关系** | ⚠️ 当前代码库中 `KnowledgeNode`/`KnowledgeEdge` 实体映射 `ecos_knowledge_graph_node` 表，通过 MyBatis CRUD + `Java stream filter` 做图遍历。引入 Neo4j 是合理的升级路径，但需注意：**不是替换 PG 表，而是新增 Neo4j 作为图计算引擎**，两者需建立数据同步机制（如 CDC/定时同步）。 |
| **风险** | Neo4j Docker 容器至少需要 1-2GB RAM（含 JVM heap + page cache），在 WSL 单机上与 PG + Java 应用 + OPA 容器共存可能导致资源争抢。 |

**兼容性结论**: ✅ 技术上兼容，但需新增 Maven 依赖并设计 PG↔Neo4j 数据同步策略。

#### 2.1.2 OPA 0.63 (Docker)

| 项目 | 评估 |
|------|------|
| **成熟度** | ⭐⭐⭐⭐⭐ OPA 是 CNCF 毕业项目，被 Kubernetes、Istio、Terraform 等广泛采用。0.63 版本（约 2024 Q4 发布）功能稳定。但截至 2026 年中，最新稳定版应为 0.70+，**建议使用 0.68+ 以获取更好的性能优化**。 |
| **Spring 兼容性** | ⭐⭐⭐⭐ OPA 提供 REST API（默认端口 8181），Spring 通过 `RestTemplate`/`WebClient` 调用，架构解耦良好。Java 侧也可使用 `opa-java` SDK 简化集成。 |
| **与现有 ABAC 的关系** | 当前 `InMemoryAbacPolicyService` 是纯内存 `ConcurrentHashMap`，无持久化、无热加载。OPA 替代后带来：Rego 策略声明式管理、Bundle 热加载、审计日志。但需设计策略迁移方案（Java 条件 → Rego 规则）。 |
| **热加载延迟** | OPA 支持 `--watch` 模式的 Bundle 热加载，延迟通常 <500ms（文件系统 watch + Rego 编译）。对 ABAC 授权场景足够。 |

**兼容性结论**: ✅ 通过。OPA 作为独立 HTTP 服务与现有 Java 栈解耦良好。

#### 2.1.3 Spring WebSocket + STOMP

| 项目 | 评估 |
|------|------|
| **成熟度** | ⭐⭐⭐⭐⭐ Spring Framework 自 4.0 内置 WebSocket 支持，STOMP 子协议自 4.0.0 就成熟。企业级消息推送的标准方案。 |
| **与现有栈兼容性** | ✅ 当前后端是 Spring Boot 3.2 + Tomcat :8080 Gateway，直接在现有 `GatewayApplication` 中启用 `@EnableWebSocketMessageBroker` 即可，无需新增端口或容器。 |
| **风险** | WebSocket 长连接管理需要：心跳保活、断线重连、Session 绑定用户身份、消息积压处理。单实例 Tomcat 下避免了分布式 Session 问题，但前端需实现 STOMP over WebSocket 客户端（React 端可用 `@stomp/stompjs`）。 |

**兼容性结论**: ✅ 完全兼容，零额外基础设施。

#### 2.1.4 Apache Commons Math3

| 项目 | 评估 |
|------|------|
| **成熟度** | ⭐⭐⭐⭐ Commons Math3 (3.6.1, 2016) 是 Apache 顶级项目，数学计算库功能全面（线性代数、统计、优化、聚类）。但已进入**维护模式**，不再新增功能。 |
| **SCM + Do-calculus 适应性** | ⚠️ Commons Math3 提供：线性代数(RealMatrix)、分布拟合、回归分析、聚类。但**不提供**结构因果模型(SCM)、Do-calculus、贝叶斯网络推断的现成实现。团队需从底层实现 SCM 数据结构 + Do-operator 图变换算法，工程量被显著低估。 |
| **替代建议** | 若需 Java 生态的因果推断库，可考虑：**Tetrad**（CMU 开发的因果发现/推断 Java 库，LGPL 许可）、**Smile**（Scala/Java ML 库含贝叶斯网络）、或通过 gRPC 调用 Python 的 `DoWhy`/`pgmpy`。Commons Math3 仅适合作为**数学底层支撑**，不能替代专用因果推断引擎。 |

**兼容性结论**: ⚠️ 需修改。Commons Math3 不足以独立支撑"SCM + Do-calculus + 贝叶斯根因反推"，建议补充 Tetrad 或明确 Commons Math3 的角色仅为数学运算层。

#### 2.1.5 pgvector 扩展

| 项目 | 评估 |
|------|------|
| **成熟度** | ⭐⭐⭐⭐⭐ pgvector 是 PostgreSQL 最流行的向量扩展，0.7.x 版本已稳定，被 Supabase、Neon 等平台内置。支持 IVFFlat/HNSW 索引，与 PG 生态无缝集成。 |
| **与现有栈兼容性** | ✅ pgvector 是 PG 扩展（`CREATE EXTENSION vector`），非独立数据库——**完全符合"不引入新数据库"原则**。当前使用 MyBatis，可通过 `PgVectorTypeHandler` 或 JSON 序列化支持向量字段。Spring AI 也提供 `PgVectorStore` 集成。 |
| **安装要求** | PG 实例需编译安装 `vector` 扩展（`apt install postgresql-XX-pgvector` 或从源码编译），需要 DBA 权限。当前 WSL 环境未确认 PG 是否已安装 pgvector。 |

**兼容性结论**: ✅ 通过，原则合规。

### 2.2 架构合理性 — 6 子项目耦合度分析

```
W7──W8──W9──W10──W11──W12──W13──W14
│                                        │
├─ P2-1: Neo4j 知识图谱 ─────────────────┤  ← 基础设施层
│  └─ 被依赖: P2-2, P2-4                │
│                                        │
├─ P2-2: Agent Mesh 真实编排 ───────────┤  ← 业务逻辑层
│  └─ 依赖: P2-1                         │
│                                        │
├─ P2-3: OPA 策略动态加载 ──────────────┤  ← 横向安全层 (独立)
│                                        │
├─ P2-4: 因果推理引擎 ──────────────────┤  ← 分析引擎层
│  └─ 依赖: P2-1, P2-5                   │
│                                        │
├─ P2-5: 自学习案例库 ──────────────────┤  ← 数据层
│                                        │
└─ P2-6: 分级告警+WebSocket ────────────┤  ← 基础设施层 (独立)
```

#### 依赖链分析

| 依赖路径 | 评估 |
|----------|------|
| P2-2 依赖 P2-1 | ⚠️ **合理但阻塞性**: Agent 编排需要知识图谱提供图遍历能力。P2-1 必须在前 2 周内完成 Neo4j 数据迁移以解除 P2-2 阻塞。 |
| P2-4 依赖 P2-1, P2-5 | ⚠️ **合理但高风险**: 因果推理需要图结构数据 + 历史案例向量。如果 P2-1 或 P2-5 延期，P2-4 直接受阻。 |
| P2-3 独立 | ✅ 完全独立，可与其他子项目并行，无耦合。 |
| P2-6 独立 | ✅ WebSocket 基础设施独立，可与任何子项目并行。 |

#### 关键架构问题

1. **P2-1 (Neo4j) 是瓶颈**: 2 个子项目直接依赖它，占据关键路径。一旦 Neo4j Docker 部署或数据迁移延期，W9 开始的 Agent Mesh 编排将无法推进。
2. **P2-5 (pgvector) 到 P2-4 (因果推理) 的依赖**: 方案中 P2-5 在 W12-14，但 P2-4 在 W11-12 就开始了——**存在时间线倒挂**。P2-4 需 P2-5 的向量检索能力，但 P2-5 还未完成。

**架构结论**: ⚠️ 需修改。P2-5 应提前到 W9-10 与 P2-3 并行，或 P2-4 推迟到 W13-14。

### 2.6 替代方案评估

| 当前选择 | 替代方案 | 优势 | 劣势 | 推荐 |
|----------|----------|------|------|------|
| Neo4j 5 Docker | **Apache AGE (PG 图扩展)** | 零额外容器，在 PG 内实现图遍历 | 社区规模小，Cypher 兼容性不如原生 Neo4j | ❌ 不推荐 (试验性) |
| Neo4j 5 Docker | **Neo4j Embedded (Java)** | 进程内运行，无 Docker 依赖 | Neo4j 5 不再推荐嵌入式模式 | ❌ 不推荐 (已弃用) |
| Neo4j 5 Docker | **PG Recursive CTE** | 纯 SQL，零引入 | 仅支持简单图遍历，无 APOC、无图算法 | ⚠️ 备选: 简单遍历可先用 CTE，复杂场景再上 Neo4j |
| Commons Math3 | **Tetrad (Java 因果推断)** | SCM、PC 算法、FCI 等开箱即用 | LGPL 许可、API 学习曲线 | ✅ **强烈推荐**: P2-4 必须用 Tetrad 替代/补充 Commons Math3 |
| Commons Math3 | **Python DoWhy + gRPC** | 业界最活跃的因果推断框架 | 引入 Python 微服务，增加部署复杂度 | ⚠️ 如 Java 方案不足，此为备选 |
| OPA Docker | **OPA Embedded (opa-java)** | 进程内调用，零延迟 | 功能不完整，部分 Rego 内置函数不可用 | ⚠️ 备选: 简单策略可嵌入，复杂场景需 Docker |
| Spring STOMP | **SSE (Server-Sent Events)** | 更简单，单向推送即够 | 不支持双向消息 | ⚠️ 如仅推送告警，SSE 更简单 |

**替代方案推荐**:
1. **P2-4 因果推理**: 将 Apache Commons Math3 作为数学底层，**新增 Tetrad** 作为 SCM + 贝叶斯网络引擎
2. **P2-6 WebSocket**: 若仅需告警推送（单向），**考虑 SSE 替代** WebSocket，前端实现更简单
3. **P2-1 Neo4j**: 无更优替代，但建议先评估能否用 **PG Recursive CTE** 满足简单图遍历场景，降级风险
