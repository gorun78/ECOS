## 一、项目定位

**ECOS**（Enterprise Cognitive Operating System）是企业级认知操作系统，核心能力：**数据治理 → 知识图谱 → 大模型Agent落地**的全链路平台。

产品分层（一套代码三套发布）：
- **标准版**: PG only（中小企业）
- **企业版**: PG + Neo4j（中型企业，因果链>3层启用图谱）
- **旗舰版**: PG + Neo4j + Doris（大型企业，单表>100万行启用列存）

## 三、架构铁律（**不可违反**）

### 3.1 DIKW分层

```
D（数据层）→ I（信息层）→ K（知识层）→ W（智能层）
     ↓              ↓              ↓            ↓
  datanet        buszhi         dccheng       aimod
  data-engine  ontology-engine  cognitive-engine
```

**依赖方向永远单向**：下层不能import上层。ArchUnit自动化测试守卫此规则。

### 3.2 四引擎架构（目标状态）

```
┌─────────────────────────────────────────────────────────┐
│                  Gateway (:8080)                        │
│  路由转发 | 认证拦截 | 安全过滤链 | 统一入口            │
├──────────┬──────────┬──────────┬───────────────────────┤
│安全引擎   │数据引擎   │本体引擎   │认知引擎               │
│认证/授权  │数据源管理  │实体建模   │Agent/Mesh            │
│审计/合规  │管道/目录   │关系/版本   │知识图谱/RAG          │
│加密/脱敏  │质量/血缘   │提案/审批   │模型/NLQ              │
│ABAC/PBAC │调度/采集   │工作流/DQ  │分类/术语              │
└──────────┴──────────┴──────────┴───────────────────────┘
```

每个引擎实现 `IEngine` 接口（`common/common-api/.../IEngine.java`）：
- `getName()` — 唯一标识
- `getStatus()` — RUNNING/DEGRADED/STOPPED
- `getConfig()` — 配置快照
- `healthCheck()` — 健康检查
- `start()` / `stop()` — 生命周期

每个引擎暴露统一端点：`/api/v1/engine/{type}/health|config|status|tasks`

### 3.3 DIKW 每层

| 层 | 模块 | 核心模型 | 数据库 |
|:--:|------|------|:--:|
| **D** 数据 | datanet / data-engine | DataSource → Pipeline → Catalog | PostgreSQL |
| **I** 信息 | buszhi / ontology-engine | ObjectType → Property → Relationship | PostgreSQL |
| **K** 知识 | dccheng + cognitive-engine | KnowledgeGraph → Classification → Glossary | PostgreSQL + Neo4j |
| **W** 智能 | aimod + cognitive-engine | Agent → Mesh → Model → NLQ | PostgreSQL |

### 3.4 GatewayApplication 扫描规则

Gateway是唯一的启动入口，通过`@ComponentScan`扫描所有模块包。**当前架构过渡期**（引擎Controller已迁移到engine/但原模块保留），通过`excludeFilters`精确排除旧模块的Controller副本，避免Bean冲突。

```java
@ComponentScan(basePackages = {
    "com.chinacreator.gzcm.gateway",
    "com.chinacreator.gzcm.common",
    "com.chinacreator.gzcm.sysman",
    "com.chinacreator.gzcm.runtime",
    "com.chinacreator.gzcm.dccheng",
    "com.chinacreator.gzcm.buszhi",
    "com.chinacreator.gzcm.aimod",
    "com.chinacreator.gzcm.market",
    "com.chinacreator.gzcm.worldmodel",
    "com.chinacreator.gzcm.workspace",
    "com.chinacreator.gzcm.portal",
    "com.chinacreator.gzcm.datanet",
    "com.chinacreator.gzcm.cognitive",
    "com.chinacreator.gzcm.engine",       // ★ 新增引擎包
}, excludeFilters = {
    // 60+ 精确排除项（Asset.ASSIGNABLE_TYPE模式）
    // 安全引擎8项 / 数据引擎5项 / 本体引擎18项 / 认知引擎15项
    // 吸收待处理: portal 5 / market 1 / worldmodel 4
})
```

**规则**：新增Controller到引擎模块时，必须在excludeFilters中同步排除旧位置副本。

### 4.4 WSL特有坑

- **UNC路径Bug**: Hermes重定向`$HOME` → Maven jansi.dll路径错误 → 用`start-gateway.sh`（含`unset HOME`）启动
- **Maven不能用Windows路径**: 必须用WSL原生`~/.local/apache-maven-3.9.11/bin/mvn`，不能用`/mnt/d/`下的
- **Git SSH过代理**: `GIT_SSH_COMMAND="ssh -o ProxyCommand='nc -X 5 -x 127.0.0.1:7897 %h %p'" git pull`
- **前端端口**: Vite dev server在WSL内启动，Windows浏览器通过localhost访问

## 五、开发规范

### 5.1 提交前必跑

```bash
bash ~/pre-check.sh
```

5道防线：编译 → TypeScript类型检查 → Gateway健康检查 → 架构门禁（ArchUnit）→ SQL语法检查

### 5.2 五不碰

1. ✋ 不改模块间依赖方向（D→I→K→W单向永不回退）
2. ✋ 不新建Maven模块不加`@ComponentScan`扫描路径（除非通过PMO指令）
3. ✋ 不改已有API路径或参数签名——只增不改
4. ✋ 不绕过`@Autowired`走`new`——`JdbcTemplate`永远通过构造器注入
5. ✋ 不产出超过10行无测试的"小型重构"——一个PR一件事

### 5.3 五允许

1. ✅ 修Bug
2. ✅ 加单元测试
3. ✅ 优化SQL
4. ✅ 补前端交互细节
5. ✅ 加日志

### 5.4 Controller开发模式

所有Controller遵循统一模式：
```java
@RestController
@RequestMapping("/api/v1/xxx")
public class XxxController {
    // 注入Service，不直接操作JdbcTemplate
    // 异常统一抛 DataBridgeException
    // 返回统一用 ApiResponse.success() / ApiResponse.badRequest() / ApiResponse.notFound()
}
```

### 5.5 前端开发模式

- 国际化：`g(locale, 'EN', '中文')` 或 `useLanguage().t()`
- 样式：用系统主题 `useTheme().styles`，禁自定义CSS
- API调用：`authHeaders()` 获取Bearer Token，`isOk()` / `payloadOf()` 解析响应
